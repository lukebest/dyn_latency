## 4.1 专家倾斜度模型

```python
import numpy as np
import matplotlib.pyplot as plt

def simulate_moe_ep_zipf(num_tokens=100000, num_experts=64, zipf_s=1.2, top_k=1):
    """
    使用 Zipf 模型模拟多 XPU 专家并行 (EP) 下的 Token 分配与发送
    
    参数:
    - num_tokens: 总共要处理和发送的 Token 数量
    - num_experts: 全局总专家数（对应切分在不同的 XPU 上）
    - zipf_s: Zipf 倾斜指数 (s 越大，热点越集中；s=0 退化为绝对均匀分布)
    - top_k: 每个 Token 选择的专家数量 (通常为 1 或 2)
    """
    print(f"=== 开始模拟 MoE 专家并行 Token 发送 ===")
    print(f"总 Token 数: {num_tokens} | 总专家数: {num_experts} | Zipf 指数 s: {zipf_s} | Top-{top_k} 路由\n")

    # 1. 构建广义 Zipf 分布的概率权重
    ranks = np.arange(1, num_experts + 1)
    weights = 1.0 / (ranks ** zipf_s)
    probabilities = weights / np.sum(weights)  # 归一化，使概率和为 1

    # 2. 模拟 Router（门控网络）为每个 Token 分配专家
    # 每个专家编号为 0 到 num_experts-1，其被选中的概率服从 Zipf 分布
    expert_ids = np.arange(num_experts)
    
    # 模拟发送：每个 Token 独立随机抽取 top_k 个不重复的专家
    allocated_tokens = []
    for _ in range(num_tokens):
        # 抽取 top_k 个专家（不允许重复）
        chosen_experts = np.random.choice(expert_ids, size=top_k, replace=False, p=probabilities)
        allocated_tokens.extend(chosen_experts)

    # 3. 统计每个专家实际接收到的 Token 数量（即 XPU 的计算负载）
    expert_loads = np.zeros(num_experts, dtype=int)
    for exp_id in allocated_tokens:
        expert_loads[exp_id] += 1

    # 4. 分析负载不均衡指标
    max_load = np.max(expert_loads)
    min_load = np.min(expert_loads)
    avg_load = np.mean(expert_loads)
    std_dev = np.std(expert_loads)
    
    # 负载不均衡度 (Imbalance Ratio): 最大负载 / 平均负载
    imbalance_ratio = max_load / avg_load if avg_load > 0 else 0

    print(f"--- 统计结果 ---")
    print(f"Top-1 超级热点专家 (ID: {np.argmax(expert_loads)}) 接收 Token 数: {max_load} (占比 {max_load/len(allocated_tokens)*100:.2f}%)")
    print(f"最冷门尾部专家 (ID: {np.argmin(expert_loads)}) 接收 Token 数: {min_load} (占比 {min_load/len(allocated_tokens)*100:.2f}%)")
    print(f"理想状态下平均每专家负载: {avg_load:.1f}")
    print(f"标准差 (各专家负载波动程度): {std_dev:.2f}")
    print(f"负载不均衡度 (Max / Avg): {imbalance_ratio:.2f}x")
    
    if imbalance_ratio > 1.5:
        print("⚠️ 警告：检测到严重的分布式热点倾斜！这将引发 All-to-All 通信时延崩塌与木桶效应。")

    return ranks, expert_loads, imbalance_ratio
’‘’
# 5 执行模拟
# 6 设定 10 万个 Token，64 个专家，倾斜指数 s=1.1 (符合自然语言长尾特征)，Top-1 路由
ranks, loads, ratio = simulate_moe_ep_zipf(num_tokens=100000, num_experts=64, zipf_s=1.1, top_k=1)
```

## 4.3 LLM多层级联简化模型

实验目的：

- 模拟 LLM 多层级联且层间数据依赖时，网络 CCT / P99 对系统级吞吐（TPS、per-NPU decode throughput）的影响
- 对比三类调度：通讯-计算串行（不掩盖）、Wide-EP TBO 掩盖、AFD microbatch ping-pong 掩盖

*模型描述（共性）*

- 模拟多 NPU 集群上 LLM 多层串行 decode：假设全体参与 NPU 同步进入 Layer 1
- 每层在逻辑上包含：Attention 计算 → Dispatch → FFN/Expert 计算 → Combine
- 各 NPU 上 Stream / 角色部署因实验而异（Wide-EP 同构 vs AFD 角色异构）
- 网络方案：§2.1 Packet Spray 或 §2.2 `ub_rg` / `ub_rg_pop`；组网：§1 场景1/2/3


### 4.3.1 系统实验1：Wide-EP Decode 通讯-计算不掩盖模型

*模型描述*

- 每个 NPU 同构部署（Wide-EP）：同时持有 Attention + 本地 Expert 分片
- 每 NPU 两个 Stream
  - Stream1（计算）：Attention、FFN/Expert GEMM
  - Stream2（通讯）：Dispatch、Combine
- 循环执行：Layer 1 → … → Layer L → 立即回到 Layer 1
- **串行依赖（本实验关键）**：同一 microbatch（此处等价于整批）上  
  `Attn → Dispatch → FFN → Combine` 四段严格串行；通讯与计算不重叠
- Dispatch 完成：本卡收齐所有源发往本卡的 token
- Combine 完成：本卡发出的 TopK 专家结果全部返回

*实验可配置参数*

| 参数 | 取值 |
| --- | --- |
| 组网场景 | 场景1、场景2、场景3 |
| 网络方案 | §2.1 Packet Spray；§2.2 `ub_rg` / `ub_rg_pop` / `ub_unscheduled` |
| NPU 同步启动 | 是 |
| EP 规模（= 参与 NPU 数） | 场景1：32 / 64 / **128**；场景2/3：256 / 512 / **1024** |
| TopK | 8 |
| Token 报文大小 | 7 KB（=1 grain，不切更小包） |
| BatchSize（每 NPU 常驻请求数） | 16，64，256，1024，4096 |
| 专家倾斜度 | Zipf，S = 0, 0.1, 0.3, 0.5, 0.7, 0.9 |
| 模型层数 L | 60（对照可扫 32 / 60 / 94，对齐 Qwen3-235B 量级） |
| Attention 计算时间 \(T_a\) | Uniform[75, 90] µs / layer / batch |
| FFN 专家计算时间 \(T_e\) | Uniform[75, 90] µs / layer / batch |

*观测指标*

1. 每层 / 每 step 墙钟时延；step 时延 \(\approx L\cdot(T_a+T_{\text{disp}}+T_e+T_{\text{comb}})\)
2. Dispatch / Combine CCT（P50/P99）、热点 vs 非热点专家时延
3. 系统级：per-NPU decode throughput \(= B / T_{\text{step}}\)


### 4.3.2 系统实验2：Wide-EP TBO（Two-Batch-Overlap）通讯计算掩盖模型

*模型描述*

- 仍为 Wide-EP 同构部署；每 NPU 两个 Stream（计算 / 通讯）同 §4.3.1
- 将全局 Batch 切成 **m=2** 个 Microbatch（MB1、MB2），形成 TBO ping-pong
- 计算 Stream 与通讯 Stream 交错：一侧算 MB\(i\) 的 Attn/FFN 时，另一侧可传 MB\(j\) 的 Dispatch/Combine
- 层内依赖仍保留（同一 MB：Attn 完成才能 Dispatch；Dispatch 齐才能 FFN；FFN 完才能 Combine），但跨 MB 可重叠

*示意调度（单 NPU，一层内片段）*

```text
Compute:  Attn-L_n-MB1 → Attn-L_n-MB2 → FFN-L_n-MB1 → FFN-L_n-MB2 → …
Comm:            Disp-L_n-MB1 → Comb←… / Disp-L_n-MB2 → …
```

*实验可配置参数*

| 参数 | 取值 |
| --- | --- |
| 组网场景 / 网络方案 / TopK / Token / Zipf / L | 同 §4.3.1 |
| Microbatch 数 m | **2**（主配置）；对照 m=1（退化为不掩盖）、m=4 |
| 每 MB 的 BatchSize | 全局 BatchSize / m；全局 BatchSize 扫 16…4096（要求能被 m 整除） |
| \(T_a,\ T_e\) | 按 **每 MB** 标定：Uniform[75, 90] µs；或按 batch 线性缩放到 \(B/m\) |

*观测与预期*

1. 相对 §4.3.1，step 时延下降；理想下通讯被计算掩盖时 \(T_{\text{step}}\) 接近 \(L\cdot\max(T_a^{\text{MB}}+T_e^{\text{MB}},\ T_{\text{net}}^{\text{MB}})\) 量级
2. 若网络 CCT 过大（Zipf 热点 / Spray 失衡），掩盖失败，P99 仍由 Dispatch/Combine 主导
3. 对比 §2.1 vs §2.2 在 TBO 下的掩盖成功率（\(T_{\text{net}} < T_{\text{compute}}\) 的比例）


### 4.3.3 系统实验3：AFD N-Microbatch 通讯计算掩盖模型

> **参考**：[FastAFD (Hao AI Lab)](https://haoailab.com/blogs/fastafd/)、[MegaScale-Infer](https://arxiv.org/abs/2504.02263)（AFD + ping-pong + M2N/N2M）。  
> 本实验把 AFD 边界映射到 §1 的 400G Clos / MpClos 超节点 fabric，用网络仿真度量 M2N/N2M 的 \(T_c\)，再用简化系统模型叠多层 decode。

#### 4.3.3.1 背景与问题（摘自 FastAFD / MegaScale-Infer）

- **Colocated MoE 张力**：长上下文下 KV 限制 decode batch，Attention 仍偏 HBM-bound，但 MoE 专家 GEMM 因每专家 token 过少而 MFU 下降；增大 EP 主要减权重流量，**不能**增大每 GPU 的 MoE token 池。
- **AFD 放置**：Attention worker 保留请求状态 / KV / Router / Sampling；FFN worker 聚合多路 Attention 的 routed token，执行 Expert，再回传层输出。记 \(M\)=Attention 侧 NPU 数，\(N\)=FFN 侧 NPU 数。
- **通讯形态变化**：层边界由对称 All-to-All 变为非对称  
  - **M2N Dispatch**：\(M\) 个 Attn → \(N\) 个 FFN  
  - **N2M Combine**：\(N\) 个 FFN → \(M\) 个 Attn  
- **Microbatch ping-pong**：全局 batch 切 \(m\) 份，使 Attn 算、FFN 算与双向通讯重叠。FastAFD 在 GB200 NVL72（极快域内互联）上测得 **mb=2 已足够隐藏双向通讯**；MegaScale-Infer 在更慢的跨节点 RDMA 上常需 **m≥3～4**。本超节点 400G Clos 介于两者之间，故将 \(m\) 作为扫描维。

#### 4.3.3.2 理想流水线模型（分析基准）

沿用 MegaScale-Infer / FastAFD 符号（单 microbatch、单层）：

| 符号 | 含义 |
| --- | --- |
| \(T_a\) | Attention 侧单 MB 计算时间 |
| \(T_e\) | FFN/MoE 侧单 MB 计算时间 |
| \(T_c\) | **单向** M2N 或 N2M 通讯时间（本实验由网络 CCT 给出；取 P99） |
| \(T_f\) | \(\max\{T_a,\ T_e\}\) |
| \(L\) | MoE 层数 |
| \(m\) | Microbatch 数 |

稳态下全局 batch 的 decode-step 上界（理想 ping-pong）：

\[
T_{\text{step}} = (T_a + T_e + 2T_c) + T_f\,(mL - 1)
\]

流水线可良好掩盖时的参考条件：

1. **计算平衡或 FFN 隐藏**：MegaScale 目标 \(T_a \approx T_e\)；FastAFD 主张更稳的工作点 \(T_e \le T_a\)（宁可有界浪费 FFN，也不要暴露 FFN 拖慢每步）
2. **单向通讯可藏**：\(T_c < T_f\)
3. **双向可藏**：\(m\,T_f \ge 2(T_f + T_c)\)（FastAFD 在快 fabric 上测得 m=2 已满足）

本仿真的核心网络问题：在场景1/2/3 + Zipf 热点下，测得的 \(T_c^{\text{M2N}}\) / \(T_c^{\text{N2M}}\) 是否仍满足 (2)(3)；若不满足，UB_RG 相对 Spray 能否把 \(T_c\) 压回可掩盖区。

#### 4.3.3.3 角色映射到本仓库组网

| 角色 | 部署 | 流量 |
| --- | --- | --- |
| Attention NPU（共 \(M\)） | 仅 Attn+Router；**不**驻留 Expert 权重 | 发 M2N routed token；收 N2M 层输出 |
| FFN NPU（共 \(N\)） | 仅 Expert 分片（EP=\(N\)） | 收 M2N；本地 Expert；发 N2M |
| 拓扑 | 场景1 单层 Clos / 场景2 Clos / 场景3 MpClos | 与 §1 相同；角色是 NPU 逻辑标签，不改物理连线 |

约束：\(M+N = P\)（\(P\) 为场景全规格 NPU 数），或子集群 \(M+N \le P\)（CI 缩减同构）。

#### 4.3.3.4 按场景重新生成的 AFD 配置

**公共参数**

| 参数 | 取值 |
| --- | --- |
| 网络方案 | §2.1 Packet Spray；§2.2 `ub_rg` / `ub_rg_pop` / `ub_unscheduled` |
| TopK | 8 |
| Token / grain | 7 KB；不切更小包 |
| 专家倾斜 | Zipf，S = 0, 0.25, 0.5, 0.75, 1.0（与全规格扫参对齐；可附加 0.1/0.3/0.7/0.9） |
| 模型层数 L | 60（主）；对照 32 / 94 |
| Microbatch m | **2**（FastAFD 默认）；对照 **1**（无掩盖）、**3**、**4**（慢网/大 \(T_c\)） |
| 每 Attn-NPU 全局 BatchSize \(B\) | 16，64，256，1024，4096（须被 m 整除） |
| 每 MB token 数 | \(B/m\)（每 Attn-NPU） |
| Attention \(T_a\) | Uniform[75, 90] µs / MB / layer（主）；可按 \(B\) 做线性缩放扫参 |
| FFN \(T_e\) | 见下方「\(T_e\) 标定」；默认使 \(T_e \le T_a\)（FastAFD 工作点）或 \(T_e \approx T_a\)（MegaScale 平衡点）两档 |
| 同步 | 全体 Attn/FFN NPU 同步进入每个 decode step |

**\(T_e\) 标定（聚合效应）**

AFD 的收益来自 FFN 侧聚合：每 FFN-NPU 每 MB 期望处理 token 量约为

\[
\text{tokens/FFN/MB} \approx \frac{M}{N}\cdot\frac{B}{m}\cdot \text{TopK}
\]

（再经 Zipf 调制到各专家）。主配置取：

- **Hidden-FFN 档**：\(T_e = 0.7\cdot T_a\)（对齐 FastAFD \(T_e\le T_a\)）
- **Balance 档**：\(T_e = T_a\)
- **Exposed-FFN 档（压力）**：\(T_e = 1.3\cdot T_a\)（故意让 FFN 暴露，观察 M:N 增大后 step 恶化）

**场景1（P=128，单层 8 平面）—— M:N 表**

> 选取可整除 128、并覆盖 FastAFD 常用「大 M、小 N」区间（文中 7:1 / 11:1 / 17:1 量级）。

#### 4.3.3.5 流量生成（M2N / N2M）

对每个 decode step、每一层、每一个 MB：

1. 每个 Attn-NPU 产生 \(B/m\) 个 token；Router（Zipf）为每个 token 抽 TopK 个 **全局专家 id**，专家均匀或按连续块映射到 N 个 FFN-NPU（EP=\(N\)）
2. **M2N**：Attn→FFN，payload = 7 KB/token；同一 (src_attn, dst_ffn) 可合并为多 grain（仿真可先 1 token=1 报文）
3. FFN 侧按专家聚合后等待 \(T_e\)（模型注入），再 **N2M** 返回等长（或按实现用 BF16 回程≈2× 体积；主配置先 **对称 7 KB** 以便与 §4.2 对比，对照档设回程 2×）
4. Attn 收齐本 MB 的 Combine 后才能进入该 MB 的下一层 Attention

#### 4.3.3.6 观测指标

1. **网络**：M2N / N2M 的 CCT P50/P99（= \(T_c\)）、平面/链路利用率、热点 FFN incast 队列
2. **掩盖判定**：\(T_c < T_f\)、\(m T_f \ge 2(T_f+T_c)\) 是否成立；失败层数占比
3. **系统**：\(T_{\text{step}}\)（仿真墙钟或解析式代入测得 \(T_c\)）、per-GPU 吞吐  
   \(\frac{M\cdot B}{(M+N)\,T_{\text{step}}}\)（FastAFD 口径：FFN NPU 计入分母但不贡献 batch）
4. **对照**：同 \(B,L,m\) 下 §4.3.1 / §4.3.2（Wide-EP）vs AFD；同 AFD 下 Spray vs `ub_rg`
5. **扫描曲线**：固定 N 扫 M（或固定比扫规模），复现 FastAFD「先升后跌」——跌点对应 \(T_e>T_a\) 或 \(T_c\) 不可藏

#### 4.3.3.7 主推荐跑法（落地优先级）

| 优先级 | 场景 | 配置 | m | 网络方案 | 目的 |
| ---: | --- | --- | ---: | --- | --- |
| P0 | 场景1 | S1-B (112:16) | 2 | Spray / `ub_rg` | 最小全规格 AFD smoke |
| P0 | 场景3 | S3 等同 S2-B + Plane-striped | 2 | Spray / `ub_rg` | MpClos 平面内 M2N |
| P1 | 场景2/3 | S2-B/C + Zipf S∈{0,0.5,1.0} | 2,4 | 三种 scheme | \(T_c\) 可掩盖边界 |
| P2 | 场景2/3 | S2-D/E + Exposed-FFN 档 | 2 | `ub_rg` | 复现 FFN-bound 吞吐跌落 |
