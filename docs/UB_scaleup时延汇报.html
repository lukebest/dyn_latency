SECTION ONE

问题定位

在 BSP 同步的 MoE 训练/推理中，网络时延的尾部（p99），而非均值，直接决定系统吞吐。

1 μs 级交付差异 → 约 50% 系统吞吐退化

BSP 步长由“最慢者”决定：均值不重要，max / p99 才是有效指标。这是本研究的出发点。

背景与命题
HUAWEI
CONTEXT

MoE Dispatch/Combine 的通信画像

BSP 同步阶段

Dispatch / Combine 是全体节点近似 lockstep 的批量 all-to-all；每一步必须“全网确知完成”才能推进。

目的批量已知

门控在一次批量计算中同时给出全部 token 的目的地 —— 阶段开始时，网络就可获知完整的 N×N 需求矩阵。

次序自由

发送方不关心这批 token 以何种顺序发出 —— 这是把“逐包在线仲裁”换成“批量编排”的关键自由度。

命题

现网自由注入分组交换把这两个结构性先验浪费了，代价是 p99 的排队长尾。本研究要把它们用起来。

有界时延分析

把 scale-up 网络 p99 端到端时延拆成四层，逐层判断：哪些是物理硬墙，哪些是可优化的调度产物。

p99 = 静态 + incast 硬墙 + 拥塞排队 + 屏障感知

结论先行：真正拖高 p99 的不是 incast 硬墙，而是 non-work-conserving 调度让瓶颈出口“本该忙却空转”。

01
静态时延

序列化×2 + 传播×2 + 交换 pipeline。单层 2 跳 ≈ 0.8 µs，两层 4 跳 ≈ 1.5 µs。与负载无关。

02
incast 串行

最热端口负载 × τ_g = König 下界。占动态时延绝大部分，∝ N、∝ 1/P，任何调度都打不破。

03
拥塞排队

超出下界的排队：HOL、反压拥塞树、ECMP 失衡、匹配非理想 κ。本研究主攻对象。

04
屏障感知

软件全对全 O(N²) / allreduce 树 + 掉队者偏斜，1.5–6 µs，随规模增长；不进数据面但进 BSP 关键路径。

核心洞察
HUAWEI
KEY INSIGHT · NON-WORK-CONSERVING

p99 长尾 = 瓶颈出口“本该忙却空转”

work-conserving 的定义是：只要有活等着，服务器就不空闲。现网自由注入名义守恒、实际非守恒 —— 三个机制让最热出口白白空转，把完成时间顶到 König 下界之上。

01
HOL 队头阻塞

单 FIFO 队头去往热点口的包，挡住其后本可发往空闲口的包 —— 空闲口被迫空转。

02
反压拥塞树

无损 CBFC 把热点回压传导给喂热点的源，FIFO 后续冷流全等热点排空 —— 冷 rank 被顶到 ≈11×（380 vs 34 µs）。

03
匹配非理想 κ

iSLIP 逐信元在线仲裁，出口有 eligible 输入却因入口争抢未匹配上 → 出口空转，κ≈1.1–1.8（两层 ECMP 更差）。

03
SECTION THREE

求解理论

既然拥塞是“对流量无知”的产物，那么让网络在发送前就掌握完整需求，就能像 TDM 一样算出无冲突次序。

用协议（授权节奏），而非时钟（TDM），实现确定性

三块理论基石：König 边着色下界、(σ,ρ) 网络演算、授权节奏即分布式虚拟时隙。

理论基石
HUAWEI
THEORY

从“排队消化冲突”到“调度消除冲突”

König 边着色下界

需求矩阵按 grain 量化 = 二部多重图，Δ 色边着色使调度长度 = 最忙端口负载。分组交换可达理想 TDM 完成时间，全程无冲突。

(σ,ρ) 网络演算

授权节奏把到达整形为 (σ,ρ) 曲线：ρ = 线速，σ = ⌈Δ/τ_g⌉+1。排队 ≤ σ 个 grain，只与延迟“差”有关，与负载、绝对延迟无关。

授权=虚拟时隙

调度器每 τ_g 每出口至多签发 1 grain 授权；“收到即发、FCFS 清空”使 grant 流成为分布式软时隙 —— 无需全局时钟。

范式对比
HUAWEI
CONTRAST

三种范式，如何取舍守恒与排队

自由注入（现网）

名义 work-conserving，但 HOL / 反压 / κ 使其实际非守恒；排队无界长尾。

全局时钟：不需要 · 排队：无界

TDM 时隙

严格非守恒（时隙固定，流量不匹配即浪费）；用全局绝对时钟换零排队，日历需重构。

全局时钟：必需 · 排队：0

Request/Grant（本方案）

授权节奏恢复出口守恒（有活必发），入口冲突降格为有界相位顺延；排队 ≤ σ 硬上界。

全局时钟：不需要 · 排队：数个 grain

04
SECTION FOUR

解决方案

Request/Grant 目的侧调度：先请求、后授权、按节奏注入，把冲突消解在网前而非网内。

批量匹配 + 授权节奏控制 + cursor 内建屏障

保持 Unified Bus 分组交换语义不变，仅新增 REQ / GNT / SYNC 三类控制报文。

协议流程
HUAWEI
PROTOCOL FLOW

从 token 到 barrier 的一次闭环

01
REQ 批量登记

发送方把 M 个 token 的 (buf, grain, dst, queue) 打包上报；末跳交换机拦截并交本机调度器。

02
匹配 + 节奏

调度器维护 pending 计数矩阵，每 τ_g 每出口至多授 1 grain；credit 窗口掩盖 RTT。

03
GNT 隐式时隙

授权按源聚批下发，路径逐 grain 钉扎（inject_port / spine）；发送方 FCFS 立即注入。

04
SYNC 内建屏障

出口实测转发计数对账（forwarded==expected），汇聚广播 SYNC，cursor 安全推进。

架构要点
HUAWEI
ARCHITECTURE

冲突分段解耦 · 屏障内建 · 全分布式

三段冲突解耦

下行口由目的侧调度器 D_j 集中精确调度；Spine 段确定性散布；上行段统计均衡 + 小缓冲收敛。

cursor 内建屏障

以出口转发计数精确判定“送达且全网知道”，SYNC 汇聚广播 O(N)，取代软件全对全 O(N²)。

分布式零协调

按出口物理属主分片，分片间几乎零通信；每分片 32–128 KB SRAM，资源随所辖出口数增长而非全网规模。
