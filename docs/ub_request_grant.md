# Unified Bus Request/Grant —— 低时延、低抖动、无排队的分组交换网络设计

本文档以 `ub_tdm.md`（Unified Bus 3.0 TDM 草案）为参照基准，但**不采用 TDM 方案**，仍然保持 Unified Bus 作为**分组交换（packet switched）协议**。设计目标是：在不引入全局同步时钟的前提下，构建一个**低时延、低排队抖动、低冲突、低排队甚至无排队**的分组交换网络。

目标部署场景：DeepSeek 类模型的 Expert Parallel（EP）部署，运行在单层或两层交换网络之上——单层网络最多 128 个 NPU 节点，两层网络最多 1024 个 NPU 节点，交换机 radix 为 128。

---

## 1. 设计理念：为什么可以消除排队与抖动

### 1.1 问题回顾

`ub_tdm.md` §1.1 已指出现有分组交换网络（UB 1.0/2.0）的四个核心问题：ECMP 多路径下入口选路次优、对 100 ns～1 μs 量级延迟极度敏感、BSP 屏障的全对全通知开销、以及超节点流量的强分阶段（"节气"）模式。TDM 方案通过全局同步时钟与静态时隙日历解决了这些问题，但它要求**全网绝对时钟同步**，工程代价高，且对流量模式变化的适应性受限于日历重构。

本方案的出发点是：**排队与抖动并非分组交换的固有属性，而是"网络对流量无知"的结果**。如果网络在数据发出之前就掌握了完整的流量需求，就可以像 TDM 一样计算出无冲突的传输次序——区别只在于，这张"日历"是**按需动态计算**的，而时隙边界由**授权（grant）报文的到达节奏**隐式确定，不需要全局时钟。

### 1.2 场景特征：DeepSeek EP 的 dispatch/combine

在 Expert Parallel 部署中，网络中有 N 个芯片（NPU），每个芯片**既是发送方又是接收方**。通信呈现如下特征：

1. **批量路由计算（batched routing）**：发送方 i 准备至多 M 个 token 待发送，这些 token 的目的地由 MoE 门控（gating）在一次批量计算中**同时**得出。也就是说，在 dispatch 开始的那一刻，发送方就已经知道全部 M 个 token 各自的目的节点与目的队列。
2. **token 尺寸小且规整**：依程序所处阶段不同，token 为 7 KB 或 14 KB（其它模型的 token 尺寸可能不同，但都相对较小）。
3. **BSP 式同步阶段**：所有发送方在大致相同的阶段执行 dispatch，属于 BSP（Bulk Synchronous Parallel）式的同步阶段——但**不是严格 lockstep**，因为系统没有全局同步时钟，各节点进入 dispatch 的时刻存在偏斜（skew）。
4. **send/receive 原语**：每次发送使用 send/receive 原语——发送方指定待发送 token 缓冲区的地址（本地 buffer id）、目的节点 id、以及**接收队列 id**（而非接收缓冲区地址），由接收方将数据落入对应队列。

### 1.3 核心洞察：发送次序的自由度

优化 dispatch 延迟之所以可能，源于一个关键事实：

> **发送方并不关心自己这 M 个 token 以什么顺序发出。**

对应用而言，只要在 BSP 阶段结束前所有 token 送达即可；token 之间没有顺序依赖。这一自由度意味着：

- 传统分组交换中，交换机面对的是**逐包在线仲裁**问题——每个包到达时才知道它要去哪，冲突只能靠排队消化，排队又造成延迟与抖动。
- 而在本场景中，全部 N 个发送方的全部需求（一个 N×N 的**流量需求矩阵** D，D[i][j] = 节点 i 发往节点 j 的数据量）在阶段开始时即可被网络获知。**通过编排每个发送方向每个接收方交付数据的次序**，可以离线/在线地求出一张**无冲突调度表**，使得 token 在交换机内部以及多级网络的整条交付路径上都不发生冲突。

**理论保证**：把需求矩阵 D 以 grain（见 §2.1）为单位量化后，它对应一张二部多重图（源端口 × 目的端口，边的重数 = grain 数）。由 König 边着色定理，二部图可用 Δ 种颜色完成边着色，Δ 为最大度数——即**调度长度恰好等于最忙端口的负载**：

> 调度长度 = max( max_i Σ_j D[i][j], max_j Σ_i D[i][j] )（以 grain 计）

这是任何网络都不可能突破的下界（最忙端口必须传完它的数据），而无冲突调度**恰好达到**该下界。换言之：**只要允许重排发送次序，分组交换可以做到与理想 TDM 完全相同的完成时间，且全程无排队冲突。** 对于均衡 all-to-all，该调度退化为 N−1 个轮转置换（rotation permutation）的序列，与 `ub_tdm.md` §4.6 的 T = N 日历完全对应。

### 1.4 Request/Grant 协议总体机制

在端点（发送方/接收方）与网络之间引入**请求/授权（request/grant）协议**：

```
 发送方 NPU                          网络（调度器）                      接收方 NPU
    │                                     │                                │
    │── Batch Request ───────────────────▶│  维护全部发送方的请求状态         │
    │   [(buffer_id, size, dst+queue),…]  │  持续更新需求矩阵                │
    │                                     │  计算最优无冲突调度              │
    │◀────────────────── Batch Grant ─────│  按调度节奏签发授权              │
    │   [buffer_id, …]                    │                                │
    │  授权入 FCFS grant 队列              │                                │
    │  立即服务队列直至清空                 │                                │
    │── Data(grain) ═════════════════════════════ 无冲突路径 ══════════════▶│
    │                                     │                          落入 queue_id
```

1. **请求（Request）**：发送方可在任意时刻向网络发出**批量请求报文**，每个批量请求包含一串三元组 `(token_buffer_id, buffer_size, destination_node_and_queue_id)`。
2. **网络调度**：网络维护来自**所有发送方**的全部请求，持续更新全局需求状态，并据此计算最优（无冲突）调度。
3. **授权（Grant）**：网络按调度节奏向发送方签发**批量授权报文**，每个批量授权包含该发送方的一串 `buffer_id` 序列。
4. **发送方服务授权队列**：发送方收到批量授权后，将授权序列压入一个 **FCFS grant 队列**，并**立即**服务该队列直至清空——依次取出被授权的 token buffer id，将对应缓冲区数据推入网络。

**grant 即隐式时隙**：调度器以固定节奏（每个 grain 时间 τ_g 每端口至多一个授权，见 §2.1）签发 grant；发送方"收到即发、FCFS 清空"的确定性行为，使得数据注入的节奏被 grant 的签发节奏所控制。于是**grant 流本身充当了分布式的虚拟时隙时钟**——无需全局时钟同步，各节点间的偏斜由交换机内很小的弹性缓冲（数个 grain，见 §3.4、§4.5）吸收。

### 1.5 与 TDM 方案的关系与差异


| 维度                  | TDM（ub_tdm.md）              | Request/Grant（本方案）                      |
| ------------------- | --------------------------- | --------------------------------------- |
| 交换范式                | 电路/时隙交换                     | **分组交换（保持不变）**                          |
| 时钟                  | 全网绝对同步时钟，硬性时隙边界             | 无全局时钟；grant 节奏形成软时隙                     |
| 调度表                 | 静态全局日历，按"节气"预配置             | 按需动态计算，请求驱动                             |
| 无冲突保证               | 日历构造保证（time_slot_tolerance） | 批量匹配/边着色构造保证 + grant 节奏控制               |
| 排队                  | 严格零排队                       | 近零排队（数个 grain 的弹性缓冲上界）                  |
| 完成时间（均衡 all-to-all） | K × EP × τ + HOP × τ        | 同一量级：max 负载 × τ_g + RTT 启动开销 + HOP 直通延迟 |
| 流量适应性               | 需重构日历                       | 天然适应任意需求矩阵（含非均衡）                        |
| 抖动                  | 0                           | 有界：≤ 弹性缓冲深度 × τ_g                       |


本质上，本方案把 TDM 的"离线全局日历"替换为"在线批量匹配 + 授权节奏控制"，用**协议**而非**时钟**实现同等的确定性。

## 2. 协议设计

### 2.1 固定 grain 粒度

若报文尺寸任意，调度问题会显著复杂化：不同长度的报文占用出口的时间不同，无冲突调度退化为带权区间调度，难以用规整的匹配算法求解，也难以让 grant 节奏与端口带宽精确对齐。

因此本协议规定：**调度以固定 grain 为最小粒度**。grain 类似 TDM 的时隙（time slot）：

- **grain 尺寸 G**：每"节气"（通信阶段）配置一次，全网一致。可选集合建议为 {1792 B, 3584 B, 7168 B, 14336 B}，以匹配典型 token 尺寸。推荐取 **G = 7168 B（7 KB）**：DeepSeek dispatch 阶段 7 KB token 恰为 1 grain，14 KB token 为 2 grain。
- **grain 时间 τ_g**：一个 grain 在链路上的串行化时间。以 400 Gb/s（≈50 GB/s 有效）链路为例，τ_g = 7168 B ÷ 50 GB/s ≈ **143 ns**。τ_g 即本系统的"虚拟时隙"长度。
- **量化规则**：长度为 L 的 token 占用 ⌈L/G⌉ 个 grain，末 grain 不足部分填充（padding）；请求中的 `grain_count` 即该值。每个 grain 作为**一个独立的数据报文**传输。
- **两类流量**：
  - **Scheduled 类**：全部 token 数据，必须先请求、后授权、再发送，按 grain 量化。
  - **Unscheduled 类**：Request/Grant/屏障通知等微小控制报文（≤ 256 B），走严格高优先级、免调度直发。由于其带宽占比极低（< 0.2%，见 §3.6），对 Scheduled 类的干扰上界为每 hop 一个控制报文的传输时间（≈ 5 ns @400G），可忽略。

### 2.2 报文类型总览


| 类型                 | 编码      | 方向           | 类别          | 作用                                                                                  |
| ------------------ | ------- | ------------ | ----------- | ----------------------------------------------------------------------------------- |
| REQ                | 0x1     | 端点 → 调度器     | Unscheduled | 批量登记发送需求                                                                            |
| GNT                | 0x2     | 调度器 → 端点     | Unscheduled | 批量签发发送授权                                                                            |
| DATA               | 0x3     | 端点 → 端点      | Scheduled   | 承载一个 grain 的 token 数据                                                               |
| REQ_ACK / GNT_NACK | 0x4/0x5 | 双向           | Unscheduled | 可靠性辅助（见 §2.8）                                                                       |
| PHASE              | 0x6     | 管理面          | Unscheduled | 节气切换 = cursor 0 推进 + 配置载荷：G、流量类别、cursor 成员表等（见 §2.9）                                |
| SYNC               | 0x7     | 调度器/汇聚点 → 端点 | Unscheduled | cursor 完成通告：宣告某 (cursor_id, global_cursor_value) 的全部数据传输结束，BSP cursor 可安全推进（见 §2.9） |


所有报文复用 UB 链路层的成帧、CRC/FEC 与链路级 credit 流控；以下格式均指链路层载荷。

#### 2.2.1 报文端点与交换机角色：多层网络扩充说明

一个自然的问题是：多层组网时，request/grant/sync 是否也是交换机之间交换的报文？多层交换机之间是否需要不同的协议报文？**答案：单层与多层使用同一套报文类型，交换机之间不需要另立协议**；需要明确的只是报文的**协议端点**与交换机的**两种角色**。

**协议端点只有两类**：

- **NPU 端点**（发送方/接收方）；
- **调度器**（网元内的协议实体）：单层 = 每台 SW 内的平面调度器；两层 = 每台 Leaf 内的目的侧调度器 D_j，以及按 cursor 选出的汇聚点 A（A 本身就是某个 D_j，见 §4.9）。

**Spine 及处于转发路径中段的 Leaf 不是任何控制报文的端点**：不解析、不修改、不维护 request/grant/cursor 协议状态。Spine 出口的无冲突由 D_j 的确定性散布保证（§4.3），Spine 自身对协议完全无感——这是本架构可扩展性的关键。

**交换机的两种角色**：

1. **透明转发者**：所有 Unscheduled 控制报文（REQ/GNT/SYNC/ACK/NACK/PHASE）与 DATA 一样，按目的 id 做确定性直通转发（高优先级），中间交换机不触碰其语义内容；
2. **协议端点（宿主调度器）**：当控制报文的目的地是本机调度器时，上送本机调度器处理。

**唯一的"调度器 ↔ 调度器"报文**是两层组网中的 LOCAL SYNC（D_j → 汇聚点 A，§4.9）——它复用 §2.9（3）的 SYNC 格式（scope = LOCAL），不引入新报文类型；路径上的 Spine 依然只做透明转发。

**跨层寻址：CTRL 寻址头（4 B）**

单层网络中控制报文的寻址是**端口隐式**的：REQ 从 NPU 端口 k 发出即到达平面 k 的调度器，GNT 反之从平面 k 下行即隐含端口 k，报文内无需目的字段。多层网络中 REQ/GNT/SYNC 需穿越 2～3 段链路，中间交换机必须知道向哪转发，故规定：**多层部署时，所有 Unscheduled 控制报文前置一个 4 B 的 CTRL 寻址头**：


| 字段          | 位宽   | 说明                                                                    |
| ----------- | ---- | --------------------------------------------------------------------- |
| dst_kind    | 1 b  | 0 = 目的为 NPU 端点；1 = 目的为网元（调度器）                                         |
| ctrl_dst_id | 12 b | 目的 id：NPU id，或网元 id（两层 = 目的 Leaf 号，即其宿主调度器 D_j；汇聚点 A 即某个 Leaf 的网元 id） |
| src_kind    | 1 b  | 源类型，同 dst_kind                                                        |
| ctrl_src_id | 12 b | 源 id：用于 ACK/NACK 回程寻址与调度器按源测量闭环延迟（§2.10）                              |
| reserved    | 6 b  | 保留                                                                    |


**控制报文的确定性转发规则**（无 ECMP 哈希，与 DATA 一致的原则）：

- dst_kind = 0（去往 NPU）：与 DATA 相同——Spine 出口 = 目的 NPU 所在 Leaf，Leaf 下行口 = 目的 NPU；
- dst_kind = 1（去往 Leaf j 的调度器）：源侧按 (ctrl_src_id, j) 确定性轮转选 Spine（打散控制面负载），Spine 出口 = Leaf j，Leaf j 识别 ctrl_dst_id 为本机后上送调度器。

单层部署可省略 CTRL 头（沿用端口隐式寻址），也可统一携带以简化两种拓扑的实现差异——**建议统一携带**：代价 4 B/控制报文，相对满批 REQ（2060 B）占比 0.2%，可忽略。

**各报文在两种拓扑中的端点与路径汇总**：


| 报文                 | 单层：端点与路径                                                | 两层：端点与路径                           |
| ------------------ | ------------------------------------------------------- | ---------------------------------- |
| REQ                | NPU → 平面调度器（1 hop，端口隐式）                                 | NPU → D_j（3 hop；Leaf_s、Spine 透明转发） |
| GNT                | 平面调度器 → NPU（1 hop）                                      | D_j → 源 NPU（3 hop 透明转发）            |
| DATA               | NPU → NPU（2 hop）                                        | NPU → NPU（4 hop）                   |
| SYNC（LOCAL）        | 平面调度器 → 全体成员（1 hop；单层以 8 平面 LOCAL + 端点 AND 实现全局，见 §3.7） | D_j → 汇聚点 A（2 hop；唯一的调度器间报文）       |
| SYNC（GLOBAL）       | —（无需）                                                   | A → 全体成员（2～4 hop）                  |
| REQ_ACK / GNT_NACK | NPU ↔ 平面调度器（1 hop）                                      | NPU ↔ D_j（3 hop 透明转发）              |
| PHASE              | 管理面 → 全体 NPU 与调度器                                       | 管理面 → 全体 NPU 与调度器                  |


### 2.3 Request 报文格式

**报文头（12 B）**


| 字段                    | 位宽   | 说明                                                                         |
| --------------------- | ---- | -------------------------------------------------------------------------- |
| type                  | 4 b  | = REQ                                                                      |
| version               | 4 b  | 协议版本                                                                       |
| src_node_id           | 12 b | 发送方节点 id（支持至 4096 节点）                                                      |
| req_seq               | 16 b | 请求批次序号（传输层可靠性用，见 §2.8；与 cursor 的层次关系见 §2.9）                                |
| entry_count           | 8 b  | 本批条目数（0～256；0 = 空请求，用作零流量成员的 barrier 到达通告，见 §2.9）                          |
| cursor_id             | 8 b  | 本批请求所属 BSP cursor（全系统 X ≤ 256 个；cursor 0 为系统相位 cursor，取代原 phase_id，见 §2.9） |
| local_cursor_value    | 16 b | 本批请求所属的本地 cursor 世代值 v（模 2^16 回绕）                                          |
| cursor_count_expected | 20 b | 该发送方在 (cursor_id, v) 内**经由本调度器**发送的 grain 总数终值（可跨多批幂等重复声明；0 = 本世代经本调度器无数据） |
| reserved              | 8 b  | 保留                                                                         |


**请求条目（每条 8 B，最多 256 条/批）**


| 字段              | 位宽   | 说明                                          |
| --------------- | ---- | ------------------------------------------- |
| token_buffer_id | 20 b | 发送方本地 token 缓冲区 id（发送方私有含义，网络不解释）           |
| grain_count     | 6 b  | 该 token 占用的 grain 数（1～63，即 ⌈buffer_size/G⌉） |
| dst_node_id     | 12 b | 目的节点 id                                     |
| dst_queue_id    | 8 b  | 目的接收队列 id（send/receive 原语中的接收队列，非地址）        |
| priority        | 2 b  | 调度优先级（预留）                                   |
| reserved        | 16 b | 保留                                          |


一个满批请求 = 12 B + 256 × 8 B = 2060 B，可登记 256 个 token（以 7 KB token 计对应约 1.75 MB 数据），控制开销约 0.11%。

### 2.4 Grant 报文格式

**报文头（12 B）**


| 字段                | 位宽   | 说明                                            |
| ----------------- | ---- | --------------------------------------------- |
| type              | 4 b  | = GNT                                         |
| version           | 4 b  | 协议版本                                          |
| scheduler_id      | 8 b  | 签发调度器 id（单层 = 交换平面号；两层 = 目的 leaf 号）           |
| gnt_seq           | 16 b | 授权批次序号                                        |
| entry_count       | 8 b  | 本批条目数（1～64）                                   |
| credit_window     | 8 b  | 允许的在途 grain 上限 C（调度器动态下发，见 §2.7、§2.10）        |
| sync_valid        | 1 b  | 捎带的 cursor 进度字段是否有效（见 §2.9（7）：仅作进度通告，非权威屏障释放） |
| sync_cursor_id    | 8 b  | 捎带：cursor id                                  |
| sync_cursor_value | 16 b | 捎带：该 cursor 已确认完成的最新 global_cursor_value      |
| reserved          | 23 b | 保留（原 phase_id 已统一入 cursor 机制，见 §2.9（6））       |


**授权条目（每条 4 B）**


| 字段              | 位宽   | 说明                                                 |
| --------------- | ---- | -------------------------------------------------- |
| token_buffer_id | 20 b | 被授权发送的缓冲区 id（回显请求中的 id）                            |
| inject_port     | 4 b  | 发送方应使用的上行端口号（0～7；单层网络中可省略，因 grant 从平面 k 到达即隐含端口 k） |
| grain_index     | 6 b  | 多 grain token 时授权到 grain 粒度（0 = 整个 token 按序全部授权）   |
| reserved        | 2 b  | 保留                                                 |


授权条目刻意**不携带发送时刻**——发送时机完全由"grant 到达 + FCFS 立即服务"决定，这正是 §1.4 所述的隐式时隙机制。多层部署时授权条目扩展 2 B 路径钉扎字段（spine_id、dst_plane，见 §5.4）。授权开销：每 grain 约 4～6 B，相对 7 KB grain 约 0.06～0.08%。

### 2.5 Data 报文格式：极简报文头

由于路径与时机已在授权阶段确定，且网内路由是**确定性**的（单层：出口 = dst_node 直连口；两层：leaf 上行口由确定性散布公式给出、spine 出口 = 目的 leaf、leaf 下行口 = 目的 NPU），数据报文头不再需要传统报文头中的大部分字段——无需 QoS/DSCP、无需 TTL、无需分片重组控制、无需传输层端口协商、无需逐包路径选择字段。

**Data 报文 = 8 B 报文头 + G 字节载荷**：


| 字段           | 位宽   | 说明                                                                                   |
| ------------ | ---- | ------------------------------------------------------------------------------------ |
| type         | 2 b  | = DATA                                                                               |
| dst_node_id  | 12 b | 目的节点 id（编址定义见 §5.1）：交换机据此做确定性查表/公式转发，是网内转发所需的**唯一**字段（单层严格成立；多层另需 1 B 路径钉扎扩展，见 §5.4） |
| dst_queue_id | 8 b  | 目的接收队列 id：接收方 NPU 据此入队                                                               |
| src_node_id  | 12 b | 源节点 id：接收方重组与统计用                                                                     |
| token_tag    | 16 b | 源侧 token 标签（源分配，通常 = token_buffer_id 低位）：与 src_node_id 联合唯一标识一个 token                |
| grain_index  | 6 b  | 本 grain 在 token 内的序号（0 起）                                                            |
| last_grain   | 1 b  | 本 grain 是否为该 token 末 grain（接收方据此判定 token 完整）                                         |
| phase_id     | 4 b  | 系统相位 = cursor 0 的 global_cursor_value 低 4 位（防跨阶段串扰，见 §2.9（6））                        |
| reserved     | 3 b  | 保留                                                                                   |


对比：典型分组网络报文头（以太 14 B + IP 20 B + UDP 8 B + 传输层/事务层 20～60 B）合计 60～100 B，且交换机需做逐包最长前缀匹配 / ECMP 哈希 / QoS 分类。本方案报文头 8 B，开销 8/7168 ≈ **0.11%**，交换机转发逻辑退化为**一次直接索引**（dst_node_id → 出口），有利于最小化交换机 pipeline 延迟。

接收侧语义：接收方按 `(src_node_id, token_tag)` 聚合 grain，收到 `last_grain` 且 grain 齐整后，将完整 token 提交至 `dst_queue_id` 指定的接收队列——与 send/receive 原语"指定队列而非地址"的语义一致。

### 2.6 端点行为规范

**发送方**

1. 路由计算完成后，将 M 个 token 的需求打包为若干 REQ 报文（每批 ≤ 256 条）立即发出；REQ 走 Unscheduled 高优先级类，不等待。
2. 每个上行端口维护一个 **FCFS grant 队列**。GNT 到达后，其条目按序压入对应端口的 grant 队列。
3. 发送引擎对每个端口独立运行：**只要 grant 队列非空即持续服务**——取队头条目，从 `token_buffer_id` 对应缓冲区取出（第 `grain_index` 个）grain，封装 DATA 报文从该端口发出；直至队列清空。发送方**不做任何额外的节奏控制**，节奏完全由 grant 到达节奏决定。
4. 发送方对同一 token 的多个 grain 按授权顺序发出；不同 token 之间的顺序完全服从 grant 顺序（这正是利用了 §1.3 的次序自由度）。

**接收方**

1. 按 §2.5 的接收语义重组并入队；由于调度保证目的端口不超订，接收方入队速率恒 ≤ 端口线速，无需深缓冲。
2. 接收队列的空间由上层（EP 框架）按 BSP 阶段预留；`dst_queue_id` 的容量在 PHASE 配置时向调度器登记，调度器保证不超发（见 §2.7 第 4 条）。

### 2.7 网络调度器行为规范

调度器是本协议的核心网元功能（位于交换机内或旁挂），行为规范如下：

1. **状态维护**：以 grain 为单位维护需求计数矩阵 pending[i][j]（源端口 i → 目的端口 j 的未授权 grain 数）。REQ 到达即累加；签发 grant 即扣减。注意：调度器**只需计数，不需缓存任何数据**——数据始终留在发送方缓冲区，这相当于把传统交换机的 VOQ 虚拟化到了源端。
2. **匹配计算**：每个 τ_g 计算一轮**匹配（matching）**：为每个空闲目的端口挑选一个有需求的源端口，且每个源端口每轮至多被选中一次。批量需求已知 + 无 HOL 阻塞（计数器天然等价于 VOQ）的条件下，迭代式匹配（iSLIP/PIM 类）即可逼近 100% 吞吐；对于阶段性 all-to-all，可直接采用轮转置换序列（等价于 König 边着色的构造解），每轮 O(1)。
3. **授权节奏（pacing）**：对每个目的端口，每 τ_g 至多签发 1 个 grain 的授权——这保证目的端口的到达速率在流体意义上恒 ≤ 线速，是"无排队"的直接来源。对每个源端口同样每 τ_g 至多授权 1 个 grain（源端口带宽约束）。
4. **在途窗口（credit window）C**：为掩盖 request→grant→data 的往返时延（RTT），调度器允许每个源端口有至多 C 个已授权未落地的 grain 在途，C = ⌈RTT/τ_g⌉ + 裕量（典型 2～8，见 §3.4/§4.5）。C 同时是交换机弹性缓冲与接收方突发的上界。调度器亦以 credit 方式跟踪各 `dst_queue_id` 的登记容量，队列将满则暂停对该队列授权。
5. **批量签发**：允许把连续多个 τ_g 的授权合并为一个 GNT 批（≤ 64 条）以摊薄报文开销，但批的**签发时刻**仍须使发送方 FCFS 服务后的注入节奏满足第 3 条的速率约束（即批间隔 = 批大小 × τ_g）。

### 2.8 可靠性（简述）

- REQ/GNT 均带批次序号；调度器对 REQ 回 REQ_ACK（可搭载在 GNT 头），发送方超时重发 REQ（幂等：调度器按 (src, req_seq) 去重）。
- GNT 丢失由发送方侧超时检测（已 ACK 的请求长期未获授权）触发 GNT_NACK 查询，调度器重发授权。
- DATA 依赖链路层 CRC/FEC 与链路级重传；端到端由接收方按 (token_tag, grain_index) 检测缺失，通过 Unscheduled 类 NACK 触发源端对缺失 grain 重新请求（重传 grain 重新走 request/grant，保持无冲突性质）。
- cursor 推进时调度器回收 (cursor_id, 过期世代值) 的残余状态（见 §2.9）；节气切换（PHASE，即 cursor 0 推进）为其特例，防止跨阶段悬挂请求。

### 2.9 BSP Cursor 机制

BSP cursor 是内置于 request/grant 协议的世代化屏障原语：把"某一批 request 的数据是否全部送达"这一事实，变成网络可以权威宣告的事件，从而让 BSP 系统的进度游标安全推进。

**（1）概念与语义**

- 全系统支持 **X 个 cursor**（X ≤ 256），以 `cursor_id` 标识。每个 cursor 是一个独立推进的**世代计数器（epoch counter）**，服务一个 BSP 同步域（例如：一个 EP 组的 dispatch 屏障、combine 屏障可各用一个 cursor）。
- 每个 cursor 在管理面配置时登记**成员集合**（参与该屏障的节点位图）与汇聚方式（§2.9（5））。
- 每个成员节点为每个 cursor 维护本地世代值 `local_cursor_value`；网络为每个 cursor 维护权威的 `global_cursor_value`：当世代 v 的全部 request 对应数据**全部传输结束**时，网络签发 SYNC(cursor_id, v)，成员收到后即可安全推进至 v+1。
- **有界超前**：允许节点在 SYNC(v) 到达前预先登记世代 v+1 的请求（超前窗口 W_c，默认 1），调度器按 (cursor_id, value) 分账，实现请求登记与屏障等待的流水重叠；数据授权是否允许跨世代由上层语义决定（严格 BSP 则 v+1 的授权在 SYNC(v) 后才开始签发）。

**（2）REQ 携带字段的语义**（字段定义见 §2.3）

- `cursor_id` + `local_cursor_value`：声明本批请求归属的 (cursor, 世代 v)，一批内所有条目同属一个世代（发送方按世代分批，天然满足）。
- `cursor_count_expected`：该发送方在 (cursor_id, v) 内**经由本调度器**发送的 grain 总数终值。由于请求按平面/目的 Leaf 散布到多个调度器（§3.3、§4.4），每个调度器只需对账自己的份额；总数可跨多个 REQ 批幂等地重复声明。批量路由计算（§1.2）保证发送方在 dispatch 开始时即知道该终值。
- **零流量成员**：对某调度器无数据的成员，发送 entry_count = 0、cursor_count_expected = 0 的空 REQ——它就是该成员的"到达屏障"通告，保证调度器不会因某成员"无话可说"而永久等待。

**（3）SYNC 报文格式（8 B）**


| 字段                  | 位宽   | 说明                                              |
| ------------------- | ---- | ----------------------------------------------- |
| type                | 4 b  | = SYNC                                          |
| version             | 4 b  | 协议版本                                            |
| scope               | 2 b  | LOCAL = 单个调度器份额完成（用于层次汇聚）；GLOBAL = 全局完成（权威屏障释放） |
| origin_id           | 8 b  | 签发者 id（调度器或汇聚点）                                 |
| cursor_id           | 8 b  | cursor 标识                                       |
| global_cursor_value | 16 b | 已完成的世代值 v：该 (cursor_id, v) 的全部 request 数据均已传输结束 |
| reserved            | 22 b | 保留                                              |


**（4）完成判定：以出口实测数据计数为准**

调度器判定 (cursor_id, v) 份额完成的条件是：**全体成员的声明均已收到，且实际转发的 grain 计数等于声明总和**：

> ∀ 成员 s：已收到 s 的 cursor_count_expected 声明，且 forwarded[s] == expected[s]

关键设计是计**转发**而非计**授权**：调度器与出口交换机同址（单层在 SW 内、两层在目的 Leaf 内），DATA grain 从其出口转发时，按 `(src_node_id, token_tag, grain_index)` 与授权记录匹配即可归账到 (cursor_id, v)——**DATA 报文头无需增加任何 cursor 字段**，保持 §2.5 的极简格式。最后一跳（SW/Leaf → NPU）由链路层无损保证；对语义要求更严格的场景，可选按接收队列聚合的末端 ACK 模式。以转发计数为准的另一重收益见 §2.10：完成判定与 grant→注入延迟完全解耦。

**（5）层次化汇聚与广播**

- **单层网络**：8 个平面调度器各自判定本平面份额完成后，各自向 128 个成员广播 scope=LOCAL 的 SYNC；端点对 8 个平面做 AND 即得全局完成。消息量 8 × 128 = O(N)，末 grain 送达后释放时延 ≈ 判定 + 1 hop 下行 ≈ 0.3～0.4 μs。
- **两层网络**：128 个目的侧调度器 D_j 完成本地份额后，向汇聚点 **A = D_(cursor_id mod 128)** 发送 LOCAL SYNC（不同 cursor 的汇聚点按 cursor_id 打散，避免热点）；A 收齐 128 份后向全部成员广播 GLOBAL SYNC。详见 §4.9。

**（6）与 phase_id、req_seq 的关系与统一**

三个序号处于三个不同层次，其中两个可以统一：


| 序号           | 层次   | 作用                                   | 结论                                         |
| ------------ | ---- | ------------------------------------ | ------------------------------------------ |
| req_seq      | 传输层  | REQ 批次的丢包检测、去重、重传（按 (src, 调度器) 通道独立） | **保留独立**：一个世代跨多个批次，重传批次不推进世代；与 cursor 语义正交 |
| cursor value | 应用层  | BSP 世代进度，数据完成性对账                     | 本机制主体                                      |
| phase_id     | 全网相位 | 节气切换、按阶段清账、配置分发                      | **统一入 cursor**：节气本质上就是一个全体节点参与的 BSP cursor |


**统一方式**：规定 **cursor 0 为系统相位 cursor**，phase_id ≡ cursor 0 的 global_cursor_value（低位截取）。原 PHASE 报文重新定义为"cursor 0 的 GLOBAL SYNC + 新阶段配置载荷（grain 尺寸 G、流量类别、cursor 成员表等）"。原"按 phase 清账"推广为按 cursor 清账：任一 cursor 推进后，调度器回收 (cursor_id, ≤ v − W_c) 的残余状态。REQ/GNT 头中的 phase_id 字段被 cursor 字段取代；DATA 头中的 phase_id 字段定义为 cursor 0 值的低 4 位，格式不变。

**（7）SYNC 捎带在 GNT 上的推演：可行但不能作为主通道**

捎带（piggyback）的好处是真实的：

- 阶段活跃期 GNT 高频流动，捎带 3～4 B（sync_valid + sync_cursor_id + sync_cursor_value，§2.4）几乎零成本、零额外报文；
- GNT 的分发路径（调度器 → 发送方）与单层 SYNC 的路径一致，活跃期的 cursor 进度可"免费"随路下发，供端点提前预取、预排下一世代。

但作为**唯一**通道存在三个根本缺陷：

- **尾部真空**：cursor 完成的时刻，恰好是该世代 pending 归零、GNT 流量枯竭的时刻——最需要投递 SYNC 时恰好没有 GNT 可捎带。若指望其它 cursor/世代的 GNT 顺路携带，则屏障释放时延取决于无关流量的有无，最坏无界。
- **覆盖不匹配**：GNT 只到达"当前有未决请求"的节点；而屏障释放必须送达**全部成员**，包括本世代零流量的成员与纯接收方节点。
- **汇聚不匹配（两层）**：GLOBAL SYNC 由汇聚点 A 签发，GNT 由各 D_j 签发；捎带需要 A → D_j → 下一个 GNT 的二次中转，路径更长、时序更复杂。

**结论：维持独立 SYNC 报文作为权威通道**（保证完成后 O(hop) 有界时延送达全体成员，Unscheduled 类微小报文的代价可忽略）；同时在 GNT 头保留捎带字段作为活跃期的进度通告与冗余加速——"独立为主、捎带为辅"。

### 2.10 授权→注入延迟与线长偏斜的应对

**问题**：发送方从收到 GNT 到 DATA 实际注入存在一步延迟

> L_i = GNT 下行传播时延 + grant 队列排队 + 本地 DMA 取数启动

其中传播项随线缆长度逐节点不同（如 5～50 m 对应约 25～250 ns 单向差异），DMA 项也有抖动。因此各发送方的"虚拟时隙"边界天然不对齐，且这种偏斜**无法靠协议消除**——追求全局对齐等于变相要求全局时钟，正是本方案要避免的。应对策略是让系统对偏斜**不敏感**，再用测量把残余偏斜**压小**：

1. **正确性不依赖对齐**。出口不超订来自速率约束（每 τ_g 每出口至多授权 1 grain，§2.7），这是流体意义的性质，对任意常量偏斜不变；偏斜只体现为弹性缓冲占用的加项 ⌈δ/τ_g⌉（§3.4、§4.5 的公式已计入），其中 δ 即各源 L_i 的展宽 + 抖动。
2. **测量与预补偿（发令提前量）**。调度器与出口交换机同址，可对每个源持续测量闭环延迟 L_i（GNT 签发时刻 → 对应 DATA 到达本机的时刻），维护 per-source 滑动平均。签发授权时对源 i 施加**提前量 (L_i − min_k L_k)**，使不同源的 grain 到达出口的时刻近似对齐——类比接力赛按道次错时鸣枪、或 SDH 的指针调整。这只需要调度器**本地**时钟做差分测量，无需任何跨节点时钟同步；预补偿后 δ 从"线长差全展宽"收敛到测量残差（数十 ns 量级），进一步压缩缓冲占用与抖动。
3. **cursor 判定与注入延迟解耦**。§2.9（4）规定完成判定以出口实测转发计数为准、而非授权签发计数，因此 grant→注入延迟只影响完成事件"何时"发生（推迟约 max L_i），不影响判定的**正确性**——线长偏斜对 BSP 语义零影响。
4. **守护定时器**。已授权 grain 超过 T_guard（≥ max L_i + 余量）未到达出口，触发 §2.8 可靠性路径（重授权/重请求），保证 cursor 完成时延即使在丢包场景下也有上界。
5. **per-source 在途窗口**。credit_window 按各源实测取 C_i = ⌈2L_i'/τ_g⌉ + 裕量（L_i' 含请求上行段），远端节点获得更大窗口以填满流水线，近端节点不多占缓冲上界。

**偏斜不敏感性的形式化论证：为什么迟到不会破坏无冲突调度**

一个自然的疑问是：grant→注入延迟是否会使报文**错过无冲突的发送窗口**，在交换机端口与链路上引发冲突，令全局无冲突调度失去意义？答案是否定的，根源在于本方案的"无冲突"是**速率型/次序型**约束，而非 TDM 那种**位置型**（绝对时刻所有权）约束。

**（1）这里没有可以"错过"的窗口**

TDM 的无冲突是位置型的：时隙 t 属于流 f，全网靠绝对时钟对齐；迟到报文占用别人的时隙即为真冲突，且因 TDM 交换机无缓冲、后续时隙各有其主，错位会**级联传播**——所以 TDM 里错过窗口是灾难。本方案则相反：grant 条目刻意不携带发送时刻（§2.4），调度器计算与签发的内容中**根本没有绝对时刻**。调度器保证的唯一硬约束是速率约束（每 τ_g 每出口至多 1 grain 的授权，§2.7），"无冲突调度"的真实含义是：*任意时间窗口内压向任何链路的负载不超过链路容量加一个小常数*。这是对时间平移不变的性质——grain 晚到只是到达序列右移，窗口计数不变。

**（2）(σ, ρ) 论证：延迟不压缩流，只有延迟"差"产生积压**

对某出口 d，设授权签发时刻为 g₁ ≤ g₂ ≤ …（间隔 ≥ τ_g），第 n 个 grain 实际到达出口的时刻为 gₙ + L(n)，L ∈ [L_min, L_max]，记展宽 Δ = L_max − L_min。则任意长度 W 的窗口内到达数满足：

> 到达数 ≤ W/τ_g + ⌈Δ/τ_g⌉ + 1

即到达过程被授权节奏整形为网络演算意义上的 (σ, ρ) 到达曲线：**长期速率 ρ = 线速，突发项 σ = 延迟展宽 ⌈Δ/τ_g⌉ + 1**。注意 σ 只取决于各源延迟的**差**，与延迟绝对值无关——若全部节点线缆等长，哪怕统一延迟 10 μs，出口到达仍是完美的每 τ_g 一个、零排队。出口是 work-conserving 的串行化服务器（服务率 = 线速 ≥ ρ），故积压恒 ≤ σ 个 grain，且瞬时"重叠"在紧随其后的空隙中自动排空：一个 grain 迟到意味着它原本的位置空了出来，守恒使积压不累积。这正是 §3.4、§4.5 缓冲上界公式中 ⌈δ/τ_g⌉ 一项的来源——所谓"冲突"已被量化为**有界的弹性缓冲占用**，而非被假设不存在。

**（3）扰动局部吸收、不级联**

分组交换机的链路本来就是"串行化服务器 + 小 FIFO"：两个 grain 同瞬间争同一链路，代价是其中一个等待 ≤ 1 grain 时间，不丢弃、不重传、不占用任何其他流的资源。其他流不拥有任何时刻，它们依赖的只是自己的速率配额未被侵占——而配额由授权节奏保证，与谁迟到无关。故扰动**局部吸收、不传播**，与 TDM 的错位级联形成本质对比。量化：τ_g ≈ 143 ns，线长差 ±100 ns + DMA 抖动几十 ns → Δ ≈ 1～2 τ_g → 每出口积压加项 1～2 grain；经第 2 条的预补偿后残差收敛至几十 ns，稳态积压回到 0～1 grain。

**（4）真正的风险场景与护栏**

上述论证假设各源 L_i 近似常量。真正的威胁是某源**突然卡住**（grant 队列长时间不服务、DMA 停顿）后又一次性倾泻——此时其延迟不再有界。护栏是 credit window：该源已授权未落地的 grain 至多 C 个，超过后调度器停止授权（credit 不归还即不再签发），"迟到大军"的规模有硬上界 C，最坏瞬时积压 = C + ⌈Δ/τ_g⌉ + 1——正是 §3.4、§4.5 缓冲公式的完整形式。因此 **C 不仅是流水线填充参数，也是正确性论证的组成部分**；guard timer（第 4 条）再对丢失兜底。

**（5）小结**

TDM 用全局时钟买到 σ = 0（严格零排队）；本方案用授权节奏买到 σ = 常数个 grain。grant→注入延迟平移每个源的流而不改变其速率，全局调度保证的速率不变式原样成立："冲突"退化为有硬上界的一两个 grain 的瞬时排队，被弹性缓冲吸收且不传播。无冲突调度承诺的完成时间（最忙端口负载 × τ_g）与吞吐（100% 线速）在延迟偏斜下不受影响，唯一代价是单 grain 时延增加一个有界的 σ × τ_g 抖动项。

## 3. 单层网络详细分析（128 NPU + 8 × SW128）

### 3.1 拓扑

与 `ub_tdm.md` §3.2 拓扑二完全相同：

- 128 个 NPU，每 NPU 8 个上行口，分别连到 8 台 SW128（radix = 128），形成 8 个**独立交换平面**。
- 每台 SW128 与全部 128 个 NPU 各 1 条链路；NPU–SW 链路共 128 × 8 = 1024 条。
- 任意 (src, dst) 经任一平面均为 2 hop：NPU → SW → NPU。

### 3.2 冲突点分析

沿数据路径逐段检查：


| 链路段              | 是否可能冲突          | 说明                                   |
| ---------------- | --------------- | ------------------------------------ |
| NPU i → SW k（上行） | 否               | 该链路仅由发送方 i 独占，注入节奏由 i 自己的 grant 队列决定 |
| SW k 内部交换        | 否（无阻塞 crossbar） | radix 128 crossbar，任意置换可同时通过         |
| SW k → NPU d（下行） | **是（唯一冲突点）**    | 同一 τ_g 内若多个源的 grain 同时去往 d，则在此出口排队   |


因此单层网络的调度问题**精确地**就是 8 个相互独立的 128 × 128 crossbar 调度问题——每个平面一个，互不耦合。

### 3.3 调度器设计

- **部署位置**：每台 SW128 内置一个平面调度器，只负责本平面 128 个端口。8 个平面完全并行、无需互通。
- **状态规模**：pending 计数矩阵 128 × 128 × 16 b = 32 KB SRAM——远小于传统交换机的报文缓冲 MMU。
- **每 τ_g 一轮匹配**：τ_g ≈ 143 ns（G = 7 KB @ 400G），对硬件而言非常宽裕（同等规模的 iSLIP 交换机在数 ns 级信元时间下都可工作）。
- **grant 下发路径**：平面 k 的调度器把 GNT 从 SW k 直接沿下行链路发给源 NPU；GNT 从端口 k 到达，发送方即知本批授权应从端口 k 注入（`inject_port` 字段在单层网络可省略）。
- **端口独立性**：NPU 的 8 个端口各有独立 grant 队列、独立被 8 个平面调度，互不干扰——不存在"两个平面同时授权争抢同一发送端口"的问题。
- **请求分发**：发送方把 M 个 token 的请求按**目的分组散布**到 8 个平面。推荐沿用 `ub_tdm.md` §3.2 的确定性公式 k = (⌊i/16⌋ + ⌊d/16⌋) mod 8（源 i、目的 d），使 all-to-all 时每个平面、每个下行口的负载天然均衡；对非均衡流量，也可按 token 粒度轮转散布以均衡热点目的的 8 条下行链路。

### 3.4 无排队性与缓冲上界

**流体意义的无排队**：调度器对每个下行口每 τ_g 至多授权 1 grain（§2.7 第 3 条），故任何时间窗口 W 内到达出口 d 的数据 ≤ W/τ_g 个 grain = 线速（形式化的 (σ, ρ) 论证见 §2.10）。排队只可能来自**异步偏斜**：

1. 各发送方无全局时钟，grain 边界不对齐，两个相邻授权的 grain 可能在出口"头尾重叠"——最多贡献 1 grain 积压；
2. request/grant/data 路径的时延波动 δ（链路仲裁、控制报文插队等），贡献 ⌈δ/τ_g⌉ grain；
3. 在途窗口 C 的瞬时聚集（最坏 C 个已授权 grain 同时压向同一出口，仅出现在窗口初始瞬间）。

**出口弹性缓冲上界**：B_out ≤ C + ⌈δ/τ_g⌉ + 1 grain。取 C = 4（单层 RTT ≈ 0.5 μs ≈ 3.5 τ_g，见 §3.5）、δ ≤ 1 τ_g，则 B_out ≤ 6 grain = 42 KB/端口。整机 128 端口仅需约 5.4 MB 弹性缓冲，且这是**硬上界**而非统计值——对比传统交换机为吸收 incast 需要数十 MB 深缓冲且仍可能丢包/PFC 风暴。

**排队抖动上界**：单 grain 经历的排队延迟 ∈ [0, B_out × τ_g] ≈ [0, 0.86 μs]，且稳态（窗口爬升结束后）实际积压收敛到 0～2 grain，抖动 ≤ ±0.3 μs。对比 ECMP 分组网络中 incast 排队可达数十 μs 且长尾无界。

### 3.5 时延分析

**首 token 时延**（阶段冷启动，示例参数：400G 链路、线缆 10 m、SW 直通 pipeline 150 ns）：


| 步骤                 | 时延                   | 说明                                                |
| ------------------ | -------------------- | ------------------------------------------------- |
| REQ 串行化 + 传播       | ≈ 70 ns              | 2060 B 批请求 @400G ≈ 41 ns + 传播 50 ns（取整段 70 ns 量级） |
| SW 直通 + 递交调度器      | ≈ 150 ns             | Unscheduled 高优先级直通                                |
| 调度器首轮匹配            | ≤ 143 ns             | 一个 τ_g 内完成                                        |
| GNT 下行（SW → NPU）   | ≈ 200 ns             | 串行化（≤ 264 B）+ pipeline + 传播                       |
| 发送方取数注入            | ≈ 100 ns             | grant 队列出队 + 本地 DMA 启动                            |
| DATA 2 hop 直通      | ≈ 143 + 150 + 100 ns | 串行化 τ_g + SW 直通 + 传播                              |
| **合计（首 grain 送达）** | **≈ 1.0～1.2 μs**     |                                                   |


其中 request→grant 往返（RTT_rg ≈ 0.5～0.6 μs ≈ 4 τ_g）**只在阶段启动时支付一次**：在途窗口 C ≥ ⌈RTT_rg/τ_g⌉ = 4 即可让后续授权与数据传输全流水，稳态吞吐达线速。

**稳态每 grain 时延**：2 hop 直通 + 出口弹性缓冲 ≈ 0.4～0.7 μs，无长尾。

### 3.6 完成时间与带宽开销（DeepSeek EP128 算例）

设每 NPU 每 dispatch 阶段发送 M = 1024 个 token-send（如 batch 128 token × top-8 专家），每 token 7 KB = 1 grain：

- 每 NPU 总量 = 1024 grain = 7 MB，均分到 8 平面 → 每端口 128 grain。
- 均衡时每平面每下行口负载 = 128 grain → 完成时间 ≈ 128 × 143 ns + RTT 启动 + 2 hop ≈ **19.6 μs**，等于端口带宽下界（7 MB ÷ 8 端口 ÷ 50 GB/s = 17.9 μs）加约 1.7 μs 协议开销。与 TDM 界 K × EP × τ + HOP × τ 同量级，差异仅为一次 RTT。
- **非均衡（热点专家）**：完成时间 = max_d(去往 d 的 grain 数/8) × τ_g + 启动开销，即由最热目的的下行带宽决定——这是物理下界，且热点**不会**像传统网络那样把拥塞扩散给别的流（无 HOL、无共享队列）。此即 `ub_tdm.md` §5 非均衡修正公式 max K × EP × τ 的对应形式。
- **控制带宽开销**：REQ 8 B/token + GNT 4 B/token = 12 B / 7168 B ≈ **0.17%**；报文头开销 0.11%；合计 < 0.3%。

### 3.7 BSP 屏障的自然内置（cursor 机制）

单层网络的 BSP 屏障直接由 §2.9 的 cursor 机制实现：dispatch 阶段绑定一个 cursor，每个 NPU 把请求连同 (cursor_id, local_cursor_value, cursor_count_expected) 按平面散布到 8 个平面调度器（对某平面无数据则发空 REQ 作为到达通告）。各平面调度器以**出口转发计数**对账（forwarded == expected 对全体成员成立）判定本平面份额完成后，向 128 个 NPU 广播 scope=LOCAL 的 SYNC；NPU 对 8 个平面做 AND 即确认全局完成、推进 cursor。

消息量 8 × 128 = O(N)，而非传统全对全通知的 O(N²)；末 grain 送达后屏障释放时延 ≈ 判定 + 1 hop SYNC 下行 ≈ 0.3～0.4 μs，且由 §2.10 第 3 条，该判定不受 grant→注入延迟与线长偏斜影响。呼应 `ub_tdm.md` §1.1 问题（3）。

## 4. 两层网络详细分析（1024 NPU + 128 Leaf + 64 Spine）

### 4.1 拓扑

与 `ub_tdm.md` §3.3 拓扑三相同：

- 1024 个 NPU，每 NPU 8 个上行口连到 8 台不同 Leaf；两层 Clos，全部 SW128。
- Leaf（L1）：128 台，每台 64 口接 NPU、64 口接 Spine。
- Spine（L2）：64 台，每台 128 口——**恰好与全部 128 台 Leaf 各 1 条链路**；对偶地，每台 Leaf 与全部 64 台 Spine 各 1 条链路。
- 任意跨 Leaf 的 (src, dst) 为 4 hop：NPU → Leaf → Spine → Leaf → NPU。

### 4.2 冲突点分析


| 链路段                     | 是否可能冲突      | 竞争者                                 |
| ----------------------- | ----------- | ----------------------------------- |
| NPU → Leaf（上行第 1 段）     | 否           | 发送方独占                               |
| Leaf_s → Spine（上行第 2 段） | **是**       | 该 Leaf 下 64 个 NPU 的、恰好选中同一 Spine 的流 |
| Spine → Leaf_d（下行第 1 段） | **是**       | 全网去往 Leaf_d 且选中该 Spine 的流           |
| Leaf_d → NPU d（下行第 2 段） | **是（主冲突点）** | 全网至多 1023 个源去往同一 NPU d              |


与单层不同，两层网络有**三个**潜在冲突段；关键设计是把三段冲突解耦为"一个主调度 + 两个确定性均衡"。

### 4.3 分层调度架构

**（1）主调度：目的侧调度器 D_j（解决 Leaf_d → NPU 下行冲突）**

- 每台 Leaf j 内置一个目的侧调度器 D_j，**拥有并调度**本 Leaf 的 64 条下行链路（到 64 个本地 NPU）。
- 发送方的 REQ 按 `dst_node_id` 所属 Leaf 直接寻址到 D_j（REQ 报文走 Unscheduled 类穿过网络，2 hop 到达）。
- D_j 维护 pending[src_npu][local_dst] 计数（1024 × 64 × 16 b = 128 KB SRAM），对每条下行链路按 §2.7 节奏授权：**每 τ_g 每下行口至多 1 grain**——这使主冲突点（下行口）严格不超订。
- GNT 由 D_j 沿网络回送源 NPU（2 hop）。

**（2）确定性 Spine 散布（解决 Spine → Leaf_d 下行冲突）**

- D_j 在签发每个 grant 时**指定该 grain 应经过的 Spine**（编码进授权条目的 spine_id 字段，发送方原样抄入 DATA 头的路径钉扎字段，见 §5.4）：对去往下行口 d 的授权序列，按 **spine = (grant 序号 + src_leaf) mod 64** 轮转。
- 效果：进入 Leaf_j 的 64 条 Spine→Leaf_j 链路被 D_j 的授权节奏**均匀充满**——D_j 全部 64 个下行口合计每 τ_g 至多授权 64 grain，轮转散布后每条 Spine→Leaf_j 链路每 τ_g 至多承载约 1 grain，恒 ≤ 线速。由于 Spine→Leaf_j 的 64 条链路全部终结于 D_j 自己管辖的 Leaf，**该段冲突被 D_j 单点完全控制**，无需跨调度器协调。

**（3）源侧均衡与本地准入（收敛 Leaf_s → Spine 上行冲突）**

- 给定 grant 中的 spine 与源 NPU 的 8 台上行 Leaf，源 NPU 的注入端口由确定性公式给出（每台 Leaf 到每台 Spine 恰 1 条链路，故 spine 确定后，"经哪台上行 Leaf"即"用哪个 NPU 端口"；取 **上行 Leaf = 8 台候选中使 (leaf_id + spine) mod 8 均衡者**，写入授权条目 `inject_port`）。
- 上行段的负载来自**多个目的调度器独立签发**的授权，理论上存在瞬时叠加：最坏情况若干 D_j 恰好同时授权同一 Leaf_s 下多个 NPU、且散布到同一 Spine 链路。定量上界：Leaf_s → Spine_x 链路的竞争者为该 Leaf 下 64 个 NPU 的在途授权中散布到 x 的部分；轮转散布使期望负载 = 总上行负载/64 ≤ 线速，瞬时突发 ≤ 各源在途窗口之和在该链路上的投影，实践中 ≤ C + O(1) grain（见 §4.5 缓冲上界）。
- 为将该上界收紧为**硬保证**，可选启用**源 Leaf 准入**：Leaf_s 对每条上行链路维护 credit 计数（容量 C_up grain），DATA 注入前在 Leaf_s 入口做一次本地核对，超出则该 grain 在 Leaf_s 停留 ≤ 数个 τ_g（利用链路级 credit 自然背压到源 NPU 的发送引擎，不丢包、不扩散）。由于期望负载恒 ≤ 线速，该背压是罕见的瞬时事件。

**小结**：三个冲突段中，最难的下行口冲突由 D_j 集中式精确调度（等价于 64 端口"虚拟输出交换机"）；Spine 下行段被 D_j 的散布公式顺带确定性解决；仅上行段保留统计均衡 + 小缓冲收敛——这与 TDM 拓扑三中 k = (g + b) mod 8 公式承担的角色一一对应，但无需全局时隙表。

### 4.4 请求/授权路径与流水

- REQ：NPU → Leaf_s → Spine → Leaf_d（D_j），3 段链路，Unscheduled 直通，≈ 0.5 μs；跨层寻址由 CTRL 头承担（dst_kind = 1、ctrl_dst_id = 目的 Leaf 号，§2.2.1），Leaf_s 与 Spine 仅透明转发。
- GNT：D_j → 源 NPU，反向 3 段，≈ 0.5 μs（CTRL 头 dst_kind = 0，按 DATA 相同规则转发）。
- **RTT_rg ≈ 1.0～1.2 μs ≈ 8 τ_g**。取在途窗口 **C = 8～10** 即可全流水掩盖 RTT；C 由 GNT 头的 credit_window 字段动态下发，调度器可在阶段初期用较大 C 加速爬升、稳态回收。
- 请求聚合优化：源 NPU 把去往同一目的 Leaf 的多个 token 请求合并为一个 REQ 批（按 dst Leaf 分桶），1024 节点场景下每 NPU 每阶段发往每个 Leaf 平均仅 1 个批请求，控制报文数 O(N × 128) 而非 O(N × M)。

### 4.5 无排队性与缓冲上界

各段积压上界（推导同 §3.4：授权节奏保证流体速率 ≤ 线速，积压仅来自偏斜 δ、在途窗口 C 与多调度器叠加）：


| 位置                 | 上界（grain）                  | 典型值（G = 7 KB） |
| ------------------ | -------------------------- | ------------- |
| Leaf_s 上行口（源侧统计段）  | C_up + ⌈δ/τ_g⌉ + 2 ≈ 12    | 84 KB/口       |
| Spine 出口（到 Leaf_d） | ⌈δ/τ_g⌉ + 2 ≈ 4（D_j 确定性控速） | 28 KB/口       |
| Leaf_d 下行口（主冲突点）   | C + ⌈δ/τ_g⌉ + 1 ≈ 11       | 77 KB/口       |


单台 SW128 全端口弹性缓冲 ≤ 128 × 84 KB ≈ 10.8 MB，仍为硬上界。端到端排队抖动上界 ≈ (12 + 4 + 11) × τ_g ≈ 3.9 μs（三段最坏同时叠加，实际互斥性强），稳态收敛到 0～3 grain（≤ 0.5 μs）。对比：无调度的 1024 节点 all-to-all 在传统 Clos 上的 incast 排队可轻易达到数十至数百 μs。

### 4.6 时延分析

**首 token 时延**（冷启动）：RTT_rg（≈ 1.1 μs）+ 发送方注入（0.1 μs）+ DATA 4 hop 直通（4 × 150 ns pipeline + τ_g + 传播 ≈ 0.9 μs）≈ **2.1～2.3 μs**，仅支付一次。

**稳态每 grain 时延**：4 hop 直通 + 弹性缓冲 ≈ 0.9～1.5 μs，有界无长尾。对比 `ub_tdm.md` §4.4 的 TDM 结果（4 slot = 4τ ≈ 0.6 μs @ τ = 143 ns）：本方案稳态传输时延多出的部分主要是直通 pipeline 与小缓冲，属同一量级；换来的是免全局时钟与任意流量矩阵适应性。

### 4.7 完成时间（DeepSeek EP1024 算例）

设每 NPU 每 dispatch 阶段发送 M = 1024 个 7 KB token（每目的平均 1 个）：

- 每 NPU 7 MB，8 上行口 → 每口 128 grain ≈ 18.3 μs 注入时间；每目的下行口收 1024 × 7 KB/1024 ≈ 7 MB/8 口同理。
- 均衡完成时间 ≈ max 端口负载 × τ_g + RTT 启动 + 4 hop ≈ 128 × 143 ns + 2.3 μs ≈ **20.6 μs**，贴近带宽下界 17.9 μs；与 TDM 界 K × EP × τ + HOP × τ 同量级。
- 非均衡时同 §3.6：完成时间 = 最热下行口负载 × τ_g + O(μs)，即物理下界；热点不扩散。

### 4.8 规模与扩展性

- **调度器状态**：每 D_j 128 KB 计数 SRAM；全网 128 个 D_j 完全并行，无中心瓶颈。调度计算每 τ_g 每下行口一次选择（64 路仲裁），硬件轻量。
- **控制平面负载**：每 D_j 每阶段收 ≤ 1024 个批 REQ（每源 NPU 一批），≈ 每 τ_g 若干条，Unscheduled 类带宽占比 < 0.3%。
- **规模上限**：radix 128 交换机两层 Clos 理论上限 = 64 下行 × 128 Leaf … 若 Leaf 全部 128 口接 NPU 侧则单层 128 节点；两层按本文 64/64 分割支撑 1024 NPU（每 NPU 8 上行）。字段位宽（dst_node_id 12 b）预留至 4096 节点，为三层扩展留余量。
- **降级行为**：链路/交换机故障时，管理面更新散布公式的可用 Spine 集合（模数从 64 降为存活数），D_j 授权节奏自动按新容量收敛——不产生丢包风暴，仅按比例降速。

### 4.9 BSP cursor 的两层汇聚

两层网络中，完成判定分散在 128 个目的侧调度器 D_j 上，屏障释放需要一次汇聚（§2.9（5））：

- **本地判定**：D_j 以本 Leaf 下行出口的转发计数对账——其份额 = 全体成员声明的、以本 Leaf 下 64 个 NPU 为目的的 grain 总数（cursor_count_expected 随 REQ 按目的 Leaf 分桶送达各 D_j；对某 Leaf 无数据的成员发空 REQ）。
- **汇聚**：D_j 判定 (cursor_id, v) 本地份额完成后，向汇聚点 **A = D_(cursor_id mod 128)** 发送 scope=LOCAL 的 SYNC（2 hop，Unscheduled 类，CTRL 头以网元 id 寻址，§2.2.1）；不同 cursor 的汇聚点按 cursor_id 打散在 128 台 Leaf 上，无单点热点。
- **广播**：A 收齐 128 份 LOCAL SYNC 后，向全部 1024 个成员广播 scope=GLOBAL 的 SYNC（2～4 hop；若交换机支持硬件多播，可降为一次注入）。
- **规模与时延**：每世代消息数 O(128 + 1024)；末 grain 送达 → 屏障释放 ≈ 本地判定 + 2 hop 汇聚 + 2～4 hop 广播 ≈ **1.0～1.5 μs**。判定基于出口实测转发计数，与 grant→注入延迟、线长偏斜无关（§2.10）。
- **控制开销**：零流量成员的空 REQ 每世代 ≤ 1024 × 128 条 12 B 报文，分摊到每 D_j 每世代约 12 KB，Unscheduled 类带宽占比可忽略。

## 5. 编址与路由

前文的协议与调度分析默认了"交换机知道往哪转发"。本章补全这一层：定义统一的 node_id 地址空间、转发表的建立方式，并回答关键问题——两个 NPU 之间存在多个端口、多条路径时，路由如何处理。核心结论：**本方案中不存在逐包路径选择；多路径的每一个自由度都由一个"知情角色"在确定的时机钉扎（pin），转发退化为纯地址算术。**

### 5.1 地址空间：统一 node_id 编址

全网使用 12 bit 扁平地址空间（4096 个地址），最高位区分两类实体：


| bit 11 | 实体           | 编码                        |
| ------ | ------------ | ------------------------- |
| 0      | NPU 端点       | 端点 id（0～2047，本文用到 0～1023） |
| 1      | 网元（调度器宿主交换机） | 网元 id                     |


地址采用**结构化编码，地址即拓扑坐标**：

- **单层网络**：npu_id = 0～127，无内部结构；网元 id = 平面号 k（0～7，即 SW k 的调度器）。
- **两层网络**：1024 个 NPU 分为 **16 组 × 每组 64 个**，npu_id = {group_id 4 b, member_id 6 b}。128 台 Leaf 编码为 {group_id 4 b, plane_id 3 b}（即 Leaf 序号 = 8·group + plane）；64 台 Spine 网元 id = 0～63。布线规则与地址严格绑定：
  - NPU (g, m) 的**端口 p** 连接 Leaf (g, p) 的**下行口 m**——即组 g 的 64 个 NPU 共享 Leaf 8g～8g+7 这 8 台"目的侧平面"；
  - Leaf 的 **spine 侧端口 s** 连接 Spine s 的 **leaf 侧端口（8g+p）**——每台 Leaf 与 64 台 Spine 各恰 1 条链路。

由此，任何转发决策都可由地址字段的算术直接导出，查表仅是实现优化而非语义必需。

### 5.2 多路径的存在形式与消解原则

两个 NPU 之间的等价路径数量可观：

- **单层**：8 条（8 个平面各一条，路径 = NPU 端口 k → SW k → 目的 NPU）；
- **两层**：源端口 a（8 选 1，等价于源侧 Leaf 选择）× Spine s（64 选 1）× 目的侧平面 p（8 选 1，等价于目的侧 Leaf / 目的 NPU 入端口选择），共 **8 × 64 × 8 = 4096 条**。

传统网络在入口用 ECMP 哈希对这类多路径做**逐包盲选**，这正是 `ub_tdm.md` §1.1 问题（1）的根源。本方案的处理原则是：**每个路径自由度有唯一的钉扎者与钉扎时机，钉扎者恰好是掌握该自由度所辖资源状态的角色**：


| 路径自由度                   | 钉扎者        | 钉扎时机                             | 钉扎依据                                                                           |
| ----------------------- | ---------- | -------------------------------- | ------------------------------------------------------------------------------ |
| 目的侧平面 p（用目的 NPU 的哪个入端口） | 发送方        | **request 时**（决定 REQ 寻址到哪个调度器 D） | 请求分桶公式，把去往同一目的的流量均分到其 8 个入端口（单层特例：p 即平面 k，公式 k = (⌊i/16⌋ + ⌊d/16⌋) mod 8，§3.3） |
| Spine s                 | 目的侧调度器 D_j | **grant 时**                      | 轮转散布（§4.3（2）），保证 64 条 Spine→Leaf_d 链路速率恒 ≤ 线速                                  |
| 源侧端口 a（= 源侧 Leaf）       | 目的侧调度器 D_j | **grant 时**（写入授权条目 inject_port）  | 确定性公式（如 a = (s + m) mod 8，§4.3（3）），均衡源上行                                       |


一旦 grant 签发，该 grain 的路径完全确定，网内任何节点都不再做路径决策。**多端口 NPU 的语义**因此是：地址在 node 级、端口不编址；8 个端口是可互换的注入资源池，由 grant 逐 grain 指派——效果类似"被调度的链路聚合（LAG）"，但没有哈希失衡，因为指派者（调度器）知道每条链路的实时占用。

### 5.3 转发表的建立方式

拓扑正则 + 地址即坐标，使得转发表**由管理面算法生成、一次下发，无需任何分布式路由协议**（无 OSPF/BGP 类邻居交换与收敛过程）：

1. **拓扑发现与校验**（PHASE 0 / 上电阶段）：链路层邻居发现（hello 交换双方 node_id 与端口号），管理面比对实际布线与 §5.1 规划的绑定关系，错连线缆直接定位报警——不是"学习"拓扑，而是"校验"拓扑。
2. **表生成**：各网元的表均为地址算术，规模极小：
  - 单层 SW k：出口 = dst_node_id（identity，128 项或纯组合逻辑）；
  - Leaf (g, p)：目的在本组（group(dst) = g 且 dst_plane = p）→ 下行口 member(dst)；否则 → spine 口 spine_id（取自报文路径钉扎字段，§5.4）；
  - Spine s：出口 = 8·group(dst) + dst_plane（由报文头两个字段算出）。
3. **故障处理**：链路/网元故障时管理面只更新**可用集合掩码**（可用 Spine 集、可用平面集）并推送到相关调度器与发送方公式，复杂度 O(网元数)；调度器授权节奏按新容量自动收敛（§4.8），无路由收敛风暴。
4. **NPU 侧无路由表**：数据端口由 grant 的 inject_port 指定；控制报文按 CTRL 头寻址（§2.2.1），单层为端口隐式。

### 5.4 路径钉扎字段：DATA 与 GNT 的多层扩展

§2.5 称 dst_node_id 是网内转发的唯一字段——这在单层网络严格成立（出口 = 目的直连口）。两层网络中，Leaf_src 需要知道 grant 钉扎的 Spine s，Spine 需要知道目的侧平面 p，而两者都不能从 dst_node_id 推出。故做如下**多层扩展**：

- **DATA 报文头扩展 1 B**（多层 = 9 B，开销 9/7168 ≈ 0.13%）：


| 字段        | 位宽   | 说明                                                     |
| --------- | ---- | ------------------------------------------------------ |
| spine_id  | 6 b  | grant 钉扎的 Spine（Leaf_src 据此选 spine 口）                  |
| dst_plane | 3 b  | grant 签发者 D_j 的平面号 p（Spine 据此定目的 Leaf；也可复用基础头 3 b 保留位） |
| reserved  | 7 b* | 保留（*若 dst_plane 复用基础头保留位，则本字节余 2 b）                    |


- **GNT 授权条目扩展 2 B**（多层 = 6 B/条）：增加 spine_id 6 b 与 dst_plane 3 b。发送方注入时把授权条目中的钉扎字段**原样抄入** DATA 头，自身不做任何选路计算。

每跳转发动作归纳（均为一次地址算术，无最长前缀匹配、无哈希）：


| 节点       | 动作                                    |
| -------- | ------------------------------------- |
| Leaf_src | 出口 = spine 口 spine_id                 |
| Spine s  | 出口 = 8·group(dst_node_id) + dst_plane |
| Leaf_dst | 出口 = 下行口 member(dst_node_id)          |

## 6. 分布式调度器：可实现性、方案与性能推演

前文多处出现的"调度器"是一个**逻辑概念**，其功能包括：维护需求账本、计算匹配、按节奏生成 grant、cursor 对账与生成 SYNC、per-source 延迟测量与预补偿、credit 管理。本章论证：这个逻辑调度器可以**全分布式**地承载在单层/两层网络的多台交换机上，各分片之间几乎零协调，且硬件实现预算非常宽松。

### 6.1 分片原则：按出口链路的物理属主分片

分片规则只有一条：**每个调度器分片只调度它物理拥有的出口链路**。


| 部署  | 分片                 | 各分片调度的资源                                                 | 分片数 |
| --- | ------------------ | -------------------------------------------------------- | --- |
| 单层  | SW k 内的平面调度器       | SW k 的 128 条下行链路                                         | 8   |
| 两层  | Leaf (g, p) 内的 D_j | 本 Leaf 的 64 条 NPU 下行链路（直接），64 条 Spine→本 Leaf 链路（经散布公式间接） | 128 |


由此全网每一条可能冲突的链路都有**恰好一个速率管治者（governor）**：


| 链路段                    | 管治者                       | 机制                              |
| ---------------------- | ------------------------- | ------------------------------- |
| NPU → Leaf/SW 上行       | 源 NPU 自身                  | grant 队列 FCFS 服务，天然 ≤ 线速        |
| Leaf_s → Spine（仅两层）    | 各 D_j 的散布公式合力 + 源 Leaf 准入 | 统计均衡 + 小缓冲收敛（§4.3（3））           |
| Spine → Leaf_d（仅两层）    | 该 Leaf 的 D_j              | 授权时 Spine 轮转钉扎（§4.3（2））         |
| Leaf/SW → NPU 下行（主冲突点） | 该出口所属分片                   | 每 τ_g 每出口 ≤ 1 grain 的授权节奏（§2.7） |


分片之间**唯一的运行时交互**是 cursor 汇聚的 LOCAL SYNC（每世代每分片 1 条，O(128) 低频事件，§4.9）；grant 生成路径上零跨分片通信。

### 6.2 为什么零协调是可行的：与集中式 crossbar 调度的本质差别

经典交换机调度（iSLIP/PIM）必须在**每个信元时间**内解一次输入×输出的**二部匹配**——输入端口与输出端口两侧的约束耦合，天然需要集中仲裁或多轮迭代协商。本方案能把调度拆成互不通信的分片，靠的是两点松弛：

1. **输出侧约束不相交**：冲突资源（出口链路）按属主划分后，各分片的约束集合互不重叠；Spine 段冲突被 D_j 的确定性散布**静态**解耦（公式代替协商）。
2. **输入侧约束被"软化"**：crossbar 中同一输入每信元只能发一个信元，是**硬**约束，必须进匹配；本方案中源端口的容量由"grant 队列 FCFS + per-source credit"执行——多个分片同时授权同一源时，后果不是冲突，而是该源 grant 队列变长、注入按序顺延，弹性缓冲的 (σ, ρ) 论证（§2.10）保证出口侧不受损。**逐时隙二部匹配被松弛为"逐出口独立仲裁 + 每源信用约束"**，这正是分组交换（有弹性缓冲）相对电路/信元交换换来的调度自由度。

因此每个分片对每个出口独立跑一个仲裁器即可，无需分片间同步，也无需分片内跨出口的联合匹配。

### 6.3 入口冲突的处理：与 iSLIP 的对照

§6.2 的"输入侧约束软化"需要正面回答一个尖锐的问题：各出口独立授权时，不同出口的 grant 可能同轮选中同一个源（入口冲突）——iSLIP 类 crossbar 调度器必须同时保证 egress 与 ingress 两侧无冲突，否则授权机会因入口争抢而作废。我们如何做到类似效果？入口冲突会不会拉低端口与链路利用率？

**（1）grant 语义的根本差别：工作票，而非时隙预约**

iSLIP 面对的是无缓冲、严格时隙化的 crossbar：传输机会就是时隙本身，output 授权的 input 若在同一时隙被其它 output 抢走，该时隙**永久作废**——所以必须用 grant-accept 多轮迭代逼近极大匹配，且迭代必须在一个信元时间（几 ns）内于芯片内部完成。本方案的 grant 不是对某个时刻的预约，而是**一张排队的工作票**：出口链路是 work-conserving 的服务器，grain 何时到达就何时发出。两个出口同轮授权同一源，后果不是时隙作废，而是第二个 grain 在源侧 FCFS 队列顺延一个 τ_g、晚一拍到达出口——这正是 §2.10 中 σ 项的组成部分，被弹性缓冲吸收。**在时隙化无缓冲交换中，入口冲突消灭吞吐；在有弹性缓冲的分组交换中，入口冲突只产生有界相位顺延，吞吐积分守恒**（任意窗口内出口发出量 = 到达量，只要缓冲 ≥ σ 且授权节奏连续，出口不饿）。

需要说明的是，这**不是**"因为 grant→注入延迟不恒定所以放弃考虑入口冲突"——更准确地说：时隙级的入口-出口联合对齐在网络尺度上物理不可行（iSLIP 的迭代在片内几 ns 一轮，跨网络迭代每轮要付一次 μs 级 RTT，而虚拟时隙只有 143 ns），因此设计从一开始就不让正确性依赖时间对齐，入口冲突被换一种方式处理，而非被忽略。

**（2）分片内：入口冲突可按构造消除，且比 iSLIP 更便宜**

iSLIP 需要在线迭代，根源是信元逐个在线到达、需求每时隙才揭晓；而本方案的需求是**批量已知**的（§1.3）。分片握有完整的 pending 矩阵，可以把它滚动分解为**置换序列**（Birkhoff–von Neumann 分解；all-to-all 时退化为轮转置换，每轮 O(1)）——每轮每个源至多被选中一次，分片内入口唯一性**按构造成立**，无需任何 grant-accept 迭代。§6.4 的"每轮每源 ≤ 1 接受阶段"是在线 RR 模式下的等效强化（分片内部两相仲裁，纯硬件、无协议代价）。单层网络中每平面一个分片、源到该平面仅一条上行，因此入口冲突可做到与 iSLIP 同等严格——且因为不需要迭代而更便宜。

**（3）跨分片：无法协调，但被 credit 变成"有界且无损"**

真正无法按构造消除的是两层网络中 128 个 D_j 独立授权同一源的情形（分片间零通信是分布式的代价）。此时的机制链条：

- **同轮撞端口的概率先天被压低**：inject_port 由 Spine 轮转公式派生，同一分片的连续授权天然散布在源的 8 个端口上；不同分片轮转相位不同，进一步打散；
- **撞上也不丢**：源端口 FCFS 串行化，顺延 grain 晚几拍到达，弹性缓冲吸收；
- **credit 封顶损害范围**：某源在某分片的在途授权 ≤ C；源被持续超订（极端如 all-pull-one）时，仲裁器跳过无 credit 的源、把出口时隙授给其它有数据的源——出口因等待慢源而空转的量被限制在每源 ≤ C 个 grain，不发生 HOL 扩散。此时源的 8 个端口本就是物理瓶颈，任何调度器（含理想集中式）都无法使其更快；credit 的作用是不让这个物理瓶颈拖累别人。

**（4）利用率的定量结论**

- **持续吞吐：不降**。出口只要有可授权需求，每 τ_g 必发一张票；grain 迟到不注销票，只顺延服务；一个阶段内出口的忙时间 = 授出 grain 数 × τ_g，与到达抖动无关。
- **完成时间：加常数尾巴，不加乘法因子**。阶段完成 ≈ 最忙端口负载 × τ_g + RTT 启动 + O((C + σ) × τ_g) 收尾（最后在途的 ≤ C 个 grain 多飞一段）。以 C = 8、σ ≈ 2 计约 1.4 μs，摊在 EP1024 算例约 18 μs 的负载上 <10%，且随阶段数据量增大而摊薄——这与 §4.7"20.6 μs 对 17.9 μs 带宽下界"的差值是同一笔账。
- **唯一真正掉利用率的情形**：出口的待发需求集中于超订的源、且无其它源可补位——这是需求矩阵本身超出 König 下界前提的物理不可行，与调度机制无关。

**（5）小结**

iSLIP 用片内 ns 级迭代在每个时隙上同时解出入口、出口两侧的匹配；本方案利用需求批量已知，在**分片内**用置换分解按构造消除入口冲突，在**分片间**用"FCFS 串行化 + credit 封顶 + 弹性缓冲吸收"把残余入口冲突从吞吐损失降格为有界延迟。放弃时隙对齐不是对入口冲突的妥协，而是把 iSLIP 的迭代协商换成了物理串行化与信用反馈——换来调度器完全分布、零协调，代价是每阶段常数级收尾开销与每口几十 KB 弹性缓冲。

### 6.4 调度算法：输入、输出与无冲突输出的生成

本节给出单个分片（以两层的 D_j 为例，单层平面调度器是其"源数 128、出口数 128、无 Spine 钉扎"的退化形式）的完整调度算法。

**输入**

| 输入 | 来源 | 内容 |
|------|------|------|
| REQ 报文 | 各源 NPU | 条目 (token_buffer_id, grain_count, dst_node_id, dst_queue_id)；头部 (cursor_id, v, cursor_count_expected, req_seq) |
| DATA 报文（旁路观测） | 本机出口转发流水线 | (src_node_id, token_tag, grain_index)——用于归账、credit 归还、延迟测量 |
| 时基 | 本地时钟 | 每 τ_g 一个调度节拍（无需与任何其它节点对齐） |
| 配置 | 管理面 | cursor 成员表、初始 credit C_s、队列容量、可用 Spine 掩码 |

**输出**

| 输出 | 去向 | 内容 |
|------|------|------|
| GNT 报文 | 各源 NPU | 授权条目 (token_buffer_id, grain_index, inject_port, spine_id, dst_plane)，按源聚批、按预补偿提前量定时签发 |
| SYNC 报文 | 汇聚点 / 全体成员 | (scope, cursor_id, global_cursor_value)，见 §6.5 |
| REQ_ACK | 各源 NPU | 可靠性回执 |

**核心状态**

```text
pending[s][d]   : 每 (源 s, 本地出口 d) 一个 FIFO，元素 = (buf_id, grain_idx, queue_id, cursor, v)
credit[s]       : 源 s 的在途 grain 信用（初值 C_s，授权时减、出口归账时还）
qcredit[q]      : 目的接收队列 q 的容量信用
rr_ptr[d]       : 出口 d 的轮转仲裁指针
spine_ptr[d]    : 出口 d 的 Spine 轮转指针（仅两层）
inflight        : 在途表，键 (s, token_tag, grain_idx) → (d, cursor, v, t_issue)
L_hat[s]        : 源 s 闭环延迟 EWMA；offset[s] = L_hat[s] − min_k L_hat[k]（预补偿提前量）
```

**算法一：REQ 摄取**（事件驱动，行到行 SRAM 读改写）

```text
procedure ON_REQ(pkt):
    if duplicate(pkt.src, pkt.req_seq): send REQ_ACK; return        # 幂等去重
    ledger.declare(pkt.cursor_id, pkt.v, pkt.src, pkt.cursor_count_expected)   # cursor 声明，见 §6.5
    for e in pkt.entries:                                           # 空 REQ（entry_count=0）只做声明
        for gi in 0 .. e.grain_count−1:
            pending[pkt.src][local(e.dst_node_id)].push(
                (e.token_buffer_id, gi, e.dst_queue_id, pkt.cursor_id, pkt.v))
    send REQ_ACK
```

**算法二：调度轮**（每 τ_g 一轮；64 个出口的仲裁器为并行硬件，互不通信）

```text
procedure SCHEDULE_ROUND():                          # 每个出口 d 并行执行
    for each egress d in parallel:
        s ← rr_scan(d)          # 从 rr_ptr[d] 起轮转扫描，取第一个满足以下全部条件的源：
                                #   pending[s][d] 非空  ∧  credit[s] > 0  ∧  qcredit[队首.queue_id] > 0
        if s = NONE: continue                        # 本轮该出口无可授权者，出口空闲
        e ← pending[s][d].pop()
        credit[s] −= 1;  qcredit[e.queue_id] −= 1
        sp ← spine_ptr[d];  spine_ptr[d] ← next_enabled(sp)          # Spine 轮转钉扎（§4.3（2））
        port ← (sp + member(s)) mod 8                                # 源侧端口公式（§5.2）
        g ← grant_entry(e.buf_id, e.grain_idx, port, sp, my_plane)
        inflight[(s, tag(e), e.grain_idx)] ← (d, e.cursor, e.v, now)
        stage_for_batch(s, g)                        # 暂存入按源分组的打包缓冲
        rr_ptr[d] ← s + 1
    FLUSH_BATCHES()   # 按源聚批（≤64 条）签发 GNT；源 s 的批的签发时刻提前 offset[s]（§2.10 预补偿），
                      # 且保证同批条目隐含的注入节奏不超过第 3 章的速率约束（批间隔 = 批大小 × τ_g）
```

**无冲突性的生成机理**——输出（grant 流）逐条对应一个 grain 的完整路径与注入节奏，三类冲突资源在算法中各有一行"守恒语句"保证不超订：

1. **出口链路（主冲突点）**：`for each egress d` 每轮**至多弹出一个** pending 条目——出口 d 的授权速率被结构性地限制在 1 grain/τ_g，任何窗口内到达 ≤ 线速 + σ（§2.10）。这不是"检查后避免"冲突，而是**按构造（by construction）不产生**冲突。
2. **Spine→Leaf 链路**：`spine_ptr[d]` 轮转使本分片 64 个出口的授权在 64 条 Spine 下行链路上均匀散布，每链路 ≤ 1 grain/τ_g（§4.3（2））。
3. **源端口/上行链路**：`credit[s]` 把任何源的在途授权封顶为 C_s；多个出口（或多个分片）同轮选中同一源时，注入由源侧 FCFS 队列串行化，越额部分只表现为有界的相位顺延，由弹性缓冲吸收（§6.2、§6.3）。可选强化：单分片内加一个"每轮每源 ≤ 1"的接受阶段（两相仲裁，类似 PIM 的 accept）或直接采用置换分解（§6.3（2）），把分片内的源侧冲突按构造消除；跨分片则依靠 credit，无需协调。

**算法三：出口归账**（DATA 从本机出口转发时旁路触发，不在数据路径上增加时延）

```text
procedure ON_DATA_EGRESS(pkt):
    rec ← inflight.pop((pkt.src_node_id, pkt.token_tag, pkt.grain_index))
    L_hat[pkt.src] ← ewma(L_hat[pkt.src], now − rec.t_issue)        # 闭环延迟测量（§2.10）
    credit[pkt.src] += 1                                            # credit 归还
    ledger.account(rec.cursor, rec.v, pkt.src_node_id)              # cursor 转发计数 +1，见 §6.5
```

守护定时器扫描 `inflight` 中超龄条目（now − t_issue > T_guard），触发 §2.8 可靠性路径，保证 credit 与 cursor 不因丢失而永久悬挂。

### 6.5 cursor 对账与 SYNC 生成算法

cursor/SYNC 是调度器的第二项逻辑功能，同样按分片分布：**每分片独立对账自己的份额，唯一的跨分片动作是把"份额完成"这一比特汇聚起来**。

**账本结构**（每分片，每活跃 (cursor_id, v) 一行）

```text
ledger[c][v]:
    declared[s]   : bool     # 成员 s 的声明是否已收到（空 REQ 也算声明）
    expected[s]   : int      # s 声明的、经本分片的 grain 总数
    forwarded[s]  : int      # 本分片出口已实际转发的、属于 (c,v) 的 s 的 grain 数
    emitted       : bool     # 本分片是否已就 (c,v) 发出 LOCAL SYNC
```

**算法四：份额对账与 LOCAL SYNC**

```text
procedure ledger.declare(c, v, s, expected_cnt):
    declared[s] ← true;  expected[s] ← expected_cnt     # 幂等：重复声明必须同值
    CHECK_COMPLETE(c, v)

procedure ledger.account(c, v, s):                      # 由 ON_DATA_EGRESS 调用
    forwarded[s] += 1
    CHECK_COMPLETE(c, v)

procedure CHECK_COMPLETE(c, v):
    if emitted: return
    if ∀ s ∈ members(c): declared[s] ∧ forwarded[s] = expected[s]:
        emitted ← true
        if self = aggregator(c): ON_LOCAL_SYNC(c, v, self)          # 汇聚点自己的份额
        else: send SYNC(scope=LOCAL, c, v) to aggregator(c)          # 唯一的分片间报文
```

两个要点：完成条件的两半缺一不可——`declared` 保证"所有成员都表过态"（防止把"还没登记"误判为"没有数据"），`forwarded = expected` 保证"表态的数据都已实际穿过出口"（以转发计数为准，对 grant→注入延迟免疫，§2.10 第 3 条）；`emitted` 位保证 LOCAL SYNC 恰好发一次（重发由汇聚点侧幂等吸收）。

**算法五：全局汇聚与 GLOBAL SYNC**（仅汇聚点 A = aggregator(c) = D_(c mod 128) 执行）

```text
procedure ON_LOCAL_SYNC(c, v, from_shard):
    done[c][v].add(from_shard)                          # 幂等集合
    if |done[c][v]| = num_shards:                       # 两层 = 128；单层由端点 AND 替代本步（§3.7）
        broadcast SYNC(scope=GLOBAL, c, v) to members(c)
        gc: 通知各分片回收 ledger[c][≤ v − W_c]          # 状态回收（§2.9（6））
```

端点侧收到 GLOBAL SYNC(c, v) 即把本地 cursor c 推进到 v+1；若已按超前窗口 W_c 预登记了 v+1 的请求，则调度器无缝续接下一世代。

### 6.6 Walk-through：一个发送方从 token 到 barrier 的全过程

以两层网络（1024 NPU）为背景，跟踪 **NPU 70** 在 dispatch 阶段（绑定 cursor 3、世代 v = 41，grain G = 7 KB，τ_g ≈ 143 ns）的完整过程。按 §5.1 编址，NPU 70 = (组 1, 成员 6)，其 8 个端口连 Leaf 8～15。

**步骤 0：路由计算产出 3 个 token**

| token | 尺寸 | 目的 | 目的坐标 | 发送方选目的侧平面 p（请求分桶公式） | 请求寻址到 |
|-------|------|------|----------|------|------|
| T1（buf 17） | 7 KB = 1 grain | NPU 300，队列 5 | (组 4, 成员 44) | p = 2 | D@Leaf 34 = (4,2) |
| T2（buf 18） | 14 KB = 2 grain | NPU 301，队列 5 | (组 4, 成员 45) | p = 2 | D@Leaf 34 |
| T3（buf 19） | 7 KB = 1 grain | NPU 837，队列 5 | (组 13, 成员 5) | p = 5 | D@Leaf 109 = (13,5) |

**步骤 1（t ≈ 0）：批量请求**。NPU 70 发出：

- REQ → D@Leaf34：头 (cursor_id=3, v=41, **cursor_count_expected=3**，即 T1 的 1 grain + T2 的 2 grain)；条目 [(buf17, 1, NPU300, q5), (buf18, 2, NPU301, q5)]；
- REQ → D@Leaf109：头 (…, expected=1)；条目 [(buf19, 1, NPU837, q5)]；
- **空 REQ**（entry_count=0, expected=0）→ 其余 126 个分片：这是 NPU 70 对 (3,41) 的"到达 barrier"表态，缺了它们，那些分片的 `declared[70]` 永远为假、全网 barrier 永不释放。

三类 REQ 均为 Unscheduled 直通，约 0.5 μs 后到达各分片。

**步骤 2：REQ 摄取**（算法一）。D@Leaf34 执行：`declared[70]=true, expected[70]=3`；`pending[70][44]` 压入 (buf17, gi0)，`pending[70][45]` 压入 (buf18, gi0)、(buf18, gi1)。

**步骤 3：仲裁**（算法二）。设本轮出口 44、45 的轮转指针都扫到源 70（若有其它源竞争同一出口，则按 rr_ptr 轮流，源 70 可能晚若干轮——这正是"次序自由"被调度器行使的地方）：

| 调度轮 | 出口 | 选中 | Spine 钉扎 | inject_port = (spine + member 6) mod 8 | 授权条目 |
|--------|------|------|-----------|------------------|----------|
| 第 r 轮 | 44 | (buf17, gi0) | spine_ptr[44] = 23 | (23+6) mod 8 = 5 | (buf17, gi0, port 5, spine 23, plane 2) |
| 第 r 轮 | 45 | (buf18, gi0) | spine_ptr[45] = 24 | (24+6) mod 8 = 6 | (buf18, gi0, port 6, spine 24, plane 2) |
| 第 r+1 轮 | 45 | (buf18, gi1) | 25 | 7 | (buf18, gi1, port 7, spine 25, plane 2) |

每条授权执行 `credit[70] −= 1`（3 条在途，C=8 充裕）、写 `inflight`。注意出口 45 两轮各授 1 grain——出口速率约束按构造成立；三个 grain 被钉到三台不同 Spine、三个不同源端口，中段与上行同样无冲突。

**步骤 4：GNT 签发与发送**。三条授权聚成一个 GNT 批发往 NPU 70（签发时刻按 offset[70] 预补偿提前）。约 0.5 μs 后 GNT 到达，NPU 70 把条目按 inject_port 压入端口 5、6、7 的 FCFS grant 队列，发送引擎立即服务：

> DATA(buf17, gi0)：头 {dst 300, q5, src 70, tag 17, gi 0, last=1, 钉扎 spine 23/plane 2} → 端口 5 → Leaf 13 → （spine 口 23）→ Spine 23 → （出口 8×4+2 = 34）→ Leaf 34 → （下行口 44）→ NPU 300 入队 q5。

每跳都是 §5.4 的一次地址算术；NPU 70 全程没有做任何选路计算，只是抄写授权。

**步骤 5：出口归账**（算法三）。DATA 穿过 Leaf 34 出口 44 时：`inflight` 弹出、`L_hat[70]` 更新、`credit[70]` 归还、`ledger[3][41].forwarded[70]` 从 0 → 1。T2 的两个 grain 陆续把它推到 3 = expected[70]。D@Leaf109 侧同理记满 1。

**步骤 6：份额完成与汇聚**（算法四、五）。当 D@Leaf34 的账本对**全部 1024 个成员**满足 `declared ∧ forwarded = expected`（多数成员是 expected=0 的空声明），它向汇聚点 A = D_(3 mod 128) = D@Leaf 3 发送 LOCAL SYNC(3, 41)。A 集齐 128 个分片的 LOCAL SYNC 后，向 1024 个成员广播 GLOBAL SYNC(3, 41)。

**步骤 7（末 grain 送达后 ≈ 1.0～1.5 μs）：barrier 释放**。NPU 70 收到 GLOBAL SYNC，把 cursor 3 推进到 42，进入下一 BSP 阶段；若它已按 W_c=1 预发了 v=42 的 REQ，调度器无缝续接。

这个例子同时展示了各机制的分工：**仲裁**只发生在出口粒度（步骤 3），**无冲突**由三行守恒语句按构造保证，**cursor** 的推进完全由出口实测转发计数驱动（步骤 5–6），与 NPU 70 何时收到 grant、线缆多长（步骤 4 的延迟）无关。

### 6.7 硬件实现方案

每个分片的微架构与资源预算（工艺假设：1 GHz 调度时钟，τ_g = 143 ns ⇒ **每调度轮 143 cycle 预算**）：


| 资源                    | 单层分片（SW k）           | 两层分片（D_j）                          |
| --------------------- | -------------------- | ---------------------------------- |
| 出口数（每轮仲裁器数）           | 128                  | 64                                 |
| 候选源数（每仲裁器扇入）          | 128                  | 1024                               |
| pending 计数 SRAM       | 128×128×16 b = 32 KB | 1024×64×16 b = 128 KB              |
| grant 在途记录（guard/测量用） | ≤ 128×C ≈ 1K 项       | ≤ 1024×C ≈ 10K 项                   |
| cursor 账本             | X × 128 成员 × 2 计数    | X × 1024 成员 × 2 计数（X=16 时约 256 KB） |
| per-source 延迟表        | 128 项                | 1024 项                             |


流水线分五级（与 §6.4～6.5 的算法一～五逐一对应），均远在预算内：

1. **REQ 摄取**：解析批条目、累加 pending、登记 cursor 声明——每 τ_g 到达的条目数 ≤ 数十，SRAM 读改写即可；
2. **仲裁**：每出口一个轮转/LRU 仲裁器挑选非空源计数器。1024:1 仲裁用两级树（32×32）流水化实现，每级数 cycle；64～128 个仲裁器并行为独立硬件单元。作为参照，iSLIP 类交换机在**数 ns** 信元时间下已商用多年，本方案 143 ns 的"信元时间"宽松 1～2 个数量级；
3. **约束核查**：per-source credit、dst_queue 容量 credit、（两层）Spine 轮转指针推进与 inject_port 公式；
4. **GNT 打包**：按源聚批（≤ 64 条/批）、填 CTRL 头与钉扎字段、注入 Unscheduled 队列；
5. **DATA 归账**：出口转发的 grain 按 (src, token_tag, grain_index) 匹配在途记录，更新 cursor 转发计数与 L_i 测量（EWMA）。

grant 生成带宽：两层分片每 τ_g 至多 64 grain 授权 × 6 B = 384 B/143 ns ≈ 2.7 GB/s 的 GNT 载荷，分摊到 64 条 Spine 链路回程，每链路 < 0.1% 带宽占用。

### 6.8 性能推演

- **调度时延**：REQ 到达 → 首个 GNT 签发 ≤ 1 个调度轮 + 流水深度 ≈ 150～300 ns，是 §3.5/§4.6 首 token 时延预算中"调度器首轮匹配"一项的兑现。
- **吞吐**：每个出口的仲裁器是 work-conserving 的——只要存在有 credit 的非空源，出口每 τ_g 必发 1 个授权，故出口利用率 → 100%，完成时间达到 §1.3 的 König 下界（最忙端口负载 × τ_g + 启动开销）。
- **源端超订场景**（多分片独立授权同一源的代价）：极端如 1023 个目的同时拉取源 0 的数据，源 0 物理上限 8 grain/τ_g，grant 队列积压使其 L_0 增大。此时受损的只有源 0 自己的流（物理瓶颈使然，任何调度器都无法改变）；关键是 **per-(分片, 源) credit C 把每分片压在该源上的在途授权限制在 C 个**，出口仲裁器跳过无 credit 的源、把时隙让给其它有数据的源——瓶颈源造成的出口空转被限制在 ≤ C 个 grain，目的口保持 work-conserving，不发生传统网络的 HOL 扩散。
- **incast 场景**（1023 → 1）：目的 NPU 的 8 个入端口分属 8 个分片（两层：8 个目的侧平面），各自独立以线速授权，目的以 8 × 线速稳定收满；没有 PFC、没有丢包重传风暴，交换机缓冲占用仍遵守 §3.4/§4.5 的硬上界。
- **规模外推**：调度轮预算随 grain 时间线性放宽、随仲裁扇入对数增长（仲裁树深度）。radix 256 交换机 + 4096 端点时，扇入 4096:1 仅使仲裁树加一级（数 cycle），pending SRAM 增至约 1 MB/分片——分布式方案的资源随**单分片所辖出口数**而非全网规模增长，这是其相对集中式调度器的根本扩展性优势。

### 6.9 可用性与状态恢复

- **故障域合一**：分片与其所辖出口链路同宿主（同一台交换机），分片失效等价于该交换机失效——不引入额外的故障域，可用性不因调度功能分布化而下降。降级路径同 §4.8/§5.3：管理面更新可用集合掩码。
- **软状态设计**：分片的 pending/在途/cursor 状态均可由端点重建——REQ 幂等可重发（§2.8），cursor 声明随 REQ 重述。替换或重启一个分片后，管理面通告受影响的源端点重发未完成请求即可恢复，无需状态迁移或检查点。
- **汇聚点失效**：cursor 汇聚点 A 由 cursor_id mod 128 选定（§4.9），失效时管理面按存活 Leaf 数重映射，D_j 重发 LOCAL SYNC（幂等）。

## 7. 对比与结论


| 指标                 | 传统分组交换（ECMP + 深缓冲） | TDM（ub_tdm.md）           | Request/Grant（本文）             |
| ------------------ | ------------------ | ------------------------ | ----------------------------- |
| 全局时钟               | 不需要                | **必需（绝对同步）**             | 不需要                           |
| 排队                 | 无界（incast 长尾）      | 0                        | 硬上界数个 grain                   |
| 抖动                 | 数十 μs 长尾           | 0                        | ≤ 数 grain × τ_g（亚 μs 稳态）      |
| 均衡 all-to-all 完成时间 | 明显高于带宽下界（冲突/HOL）   | K × EP × τ + HOP × τ（下界） | 下界 + 一次 RTT（≈ +1～2 μs）        |
| 流量适应性              | 好但性能差              | 需重构日历                    | 任意需求矩阵在线适应                    |
| 报文头开销              | 60～100 B/包         | 无包头（时隙即寻址）               | 8 B/grain（≈ 0.11%）            |
| 交换机缓冲              | 数十 MB MMU          | 管线级（1～2 slot）            | 每口数十 KB 弹性缓冲                  |
| BSP 屏障             | O(N²) 全对全通知        | 日历内置                     | cursor + SYNC 汇聚广播，O(N)（§2.9） |


**结论**：利用超节点 EP 流量的两个结构性特征——**目的地批量已知**与**发送次序自由**——请求/授权协议把分组交换网络的调度从"逐包在线仲裁 + 排队消化冲突"转变为"批量匹配 + 授权节奏控制"。由 König 边着色定理保证，无冲突调度的长度恰等于最忙端口负载，即分组交换在此场景下可达到与理想 TDM 相同的完成时间界；而 grant 节奏充当分布式虚拟时隙，免去了全局时钟同步。单层 128 NPU 网络中冲突点唯一（交换机下行口），每平面一个独立调度器即可实现硬上界数个 grain 的近零排队；两层 1024 NPU 网络中，目的侧调度器集中控制主冲突点、确定性 Spine 散布消解中间段冲突、源侧小缓冲收敛残余统计波动，端到端排队与抖动同样保持亚微秒级硬上界，完成时间贴近端口带宽物理下界。

## 8. 与传统自由注入 UB 网络的定量对比

本章把 UB-Request/Grant 与**传统 UB 分组网络**做一次聚焦对比。传统机制指：每个数据包自带目的地址，发送方**自由注入**（路由计算完成即刻上线发送），交换机对逐包在线仲裁（VOQ + iSLIP 类交换调度），冲突靠交换机排队消化，端到端完成靠软件屏障感知。对比覆盖机制差异、每个 cursor step 的总 dispatch 延迟、延迟抖动与拖尾的性能预期，并在一个具体场景下量化 min/mean/max。

### 8.1 机制差异

| 维度 | 传统 UB（自由注入） | UB-Request/Grant |
|------|--------------------|------------------|
| 注入时机 | 路由计算完成即注入，无准入控制 | 先请求、后授权、按 grant 节奏注入 |
| 冲突解决位置 | 网内：交换机队列 + 在线仲裁 | 网前：调度器按构造不产生冲突（§6.4） |
| 队列位置与深度 | 交换机 VOQ/MMU，深度随流量矩阵波动，无上界 | 源端缓冲区（数据不动）+ 交换机 σ 级弹性缓冲（硬上界） |
| 逐包决策 | 每包查表/哈希选路、QoS 分类 | 路径在 grant 时钉扎，转发为一次地址算术（§5.2） |
| 屏障（cursor step 完成感知） | 软件实现：全对全通知 O(N²) 或 allreduce 树 | cursor + SYNC，出口转发计数精确判定（§6.5） |
| 新增固定步骤 | 无 | request→grant 一次 RTT（本章 §8.4 专门讨论） |

### 8.2 定性性能预期（每 cursor step）

- **首 token 延迟**：传统方案更低（≈ 0.4 μs，即刻注入 + 2 hop）；R/G 首 token 需先付 request→grant 往返（≈ 1.0～1.2 μs，§3.5）。这是传统方案唯一的结构性优势。
- **单 token 延迟抖动**：传统方案中 token 延迟取决于它在热点队列中的排队位置，本场景下为 0～2.6 μs 的宽分布；热点相关时无界拖尾。R/G 中每个 grain 的排队 ≤ σ ≈ 1～2 grain，抖动 ≤ ±0.3 μs，到达时刻近似由 grant 节奏确定。
- **step 完成时间**：两方案都受同一物理下界（最热端口负载 × τ_g）支配；差异在于传统方案叠加拥塞交互放大（HOL、背压、交换调度效率），R/G 叠加一次固定 RTT。
- **拖尾**：BSP 步时间由最慢者决定，mean 不重要、max 才重要。传统方案的 max 由"负载随机性 × 拥塞放大 × 屏障掉队者"三者相乘；R/G 的 max 几乎只剩负载随机性一项，其余是常数。

### 8.3 定量场景：128 发送方 × 64 token、随机目的、单层网络

**场景与假设**

- 拓扑：§3.1 单层网络（128 NPU + 8 × SW128），400 Gb/s 链路，G = 7 KB = 1 grain，τ_g ≈ 143 ns；
- 负载：每发送方 64 个 token（各 1 grain），目的在 128 个节点中均匀随机、动态确定（每 step 重新抽取）；总量 8192 grain；
- 对传统方案采取**优待假设**：交换机为理想 VOQ + iSLIP（输出忙则必发）、缓冲充足无丢包无 PFC 风暴——即只计入无法回避的代价。

**负载随机性（两方案共享）**

每条下行链路（共 1024 条）的负载 L ~ Binomial(1024, 1/128)：均值 8 grain，标准差 ≈ 2.8。全网最热下行口 L\* 的统计（1024 条链路取 max）：

> E[L\*] ≈ 18～19 grain；较好情形（p5）≈ 16；较差情形（p99）≈ 22

注意：**L\* 由需求矩阵与选路散布共同决定，R/G 并不能降低 L\***（单层中平面归属由请求分桶确定，与传统哈希的散布统计相同）；R/G 消除的是 L\* 之外的一切排队、放大与不确定性。

**时间模型**

| 分量 | R/G | 传统 |
|------|-----|------|
| 启动 | RTT_rg ≈ 0.6 μs（一次性） | 0（即刻注入） |
| 热口串行化 | L\* × τ_g（work-conserving，无放大） | L\* × τ_g × κ，κ = 拥塞交互放大系数（iSLIP 非均匀效率、瞬时 HOL、背压间歇；优待假设下 κ ≈ 1.0～1.5） |
| 末 grain 传输 | ≈ 0.4 μs（2 hop + σ） | ≈ 0.4 μs |
| 屏障感知 | cursor SYNC ≈ 0.4 μs（硬件，精确） | 软件屏障 ≈ 1.5～3 μs（全对全通知或 allreduce 树，含掉队者偏斜） |

**结果：一个 dispatch 轮（cursor step）的耗时**

先看**数据全部落地**（不含屏障感知）：

| 指标 | R/G | 传统 | 说明 |
|------|-----|------|------|
| min（L\*=16, κ=1） | ≈ 3.3 μs | ≈ **2.7 μs** | 传统免 RTT，纯数据落地更快 |
| mean（L\*=18.5） | ≈ 3.7 μs | ≈ 3.3 μs（κ≈1.1） | 差距 = RTT − 放大损耗 |
| max/p99（L\*=22） | ≈ 4.4 μs | ≈ 4.8 μs（κ≈1.4） | 放大系数开始反超 RTT |

再看**完整 cursor step**（数据落地 + 全网确知完成，即 BSP 语义下真正可推进的时刻）：

| 指标 | R/G | 传统 | R/G 净收益 |
|------|-----|------|-----------|
| **min** | ≈ **3.7 μs** | ≈ 4.2 μs（屏障取最优 1.5 μs） | ≈ 0.5 μs |
| **mean** | ≈ **4.0 μs** | ≈ 5.3 μs（屏障 2 μs） | ≈ 1.3 μs |
| **max / p99** | ≈ **4.8 μs** | ≈ 7～8 μs（κ≈1.4 + 屏障掉队 3 μs） | ≈ 2～3 μs |
| step 间标准差（抖动） | ≈ 0.2～0.3 μs（仅负载随机性） | ≈ 1～1.5 μs（负载 × 拥塞 × 屏障三重随机） | ~5× |

### 8.4 关于"request/grant 是新增固定步骤"的辩析

这个直觉是对的，也必须诚实承认：**R/G 每 step 新增约 0.5～0.6 μs 的一次性请求-授权往返**，纯看"数据落地"时刻，传统方案在幸运情形（min）下确实更快。但有三点使天平整体倒向 R/G：

1. **cursor step 的完整语义是"送达 + 全网知道送达"**。R/G 的屏障是协议内建的（出口计数精确判定 + SYNC 广播，0.3～0.4 μs）；传统方案因目的动态确定，接收方甚至不知道自己该收多少，必须依赖发送方尾标 + 全对全通知或 allreduce，成本 1.5～3 μs 且含掉队者偏斜。**request/grant 的固定成本被屏障侧的节省抵消而有余**（净赚约 1～2.5 μs/step）。
2. **BSP 下 max 才是有效指标**。步时间由最慢者决定：传统 p99 ≈ 7～8 μs 对 R/G ≈ 4.8 μs，每步省 2～3 μs。对照 `ub_tdm.md` §1.1（2）——1 μs 级的交付时间差异即可造成约 50% 的系统吞吐退化——这个差距在 TPOT SLA 下是决定性的。
3. **本场景对传统方案是最友好的**。均匀随机目的 + 理想交换机 + 充足缓冲，κ 才只有 1.1～1.5。真实 EP 流量的 gating 是**非均匀**的（热点专家负载可达均值的 2～5 倍）：传统方案的队列深度、缓冲占用、背压级联随热度**超线性**恶化，拖尾不可控；R/G 则线性退化到物理下界（最热口负载 × τ_g），缓冲上界与抖动界**不随热度改变**。差距随流量偏斜快速拉大。

另有一处对称的诚实说明：对**零星小消息**（不成批、单次往返语义）而言，R/G 的 RTT 摊不薄，确实不划算——本协议对这类流量本就不调度（Unscheduled 类直发，§2.1），两种机制在协议内各得其所。

### 8.5 小结

| 结论 | 内容 |
|------|------|
| 固定开销 | R/G 多付一次 RTT（≈0.6 μs），但屏障内建反赚 1～2.5 μs——完整 cursor step 的固定开销 R/G 更低 |
| mean | R/G 快约 25%（4.0 vs 5.3 μs），主要来自屏障与拥塞放大的消除 |
| max/p99（BSP 有效指标） | R/G 快约 40%（4.8 vs 7～8 μs），且方差小约 5 倍 |
| 均匀→热点流量外推 | 传统方案超线性恶化（队列/背压/缓冲随热度增长）；R/G 线性贴近物理下界，排队与抖动上界不变 |
| 传统方案保留的优势 | 首 token 延迟（≈0.4 vs ≈1.0 μs）与零星小消息——后者在本协议中归入 Unscheduled 类，两机制并存 |

### 8.6 第二组对比：每发送方 16 token（短步长场景）

将每发送方的 token 数从 64 降到 16（总量 2048 grain），其余条件与 §8.3 相同。这一组考察**串行化时间缩短、固定开销权重上升**时两方案的相对表现。

**负载随机性**

每条下行链路负载 L ~ Binomial(2048, 1/1024)：均值 **2** grain，标准差 ≈ 1.4。1024 条链路取 max：

> E[L\*] ≈ 8 grain；较好情形（p5）≈ 7；较差情形（p99）≈ 10

注意小样本统计的一个重要现象：**相对不均衡显著加剧**——L\*/均值 = 8/2 = **4 倍**（64 token 场景为 18.5/8 ≈ 2.3 倍）。总量降为 1/4，最热口负载只降为约 43%（8 对 18.5）：随机热点在小批量下相对更"热"，这对两个方案同等生效。

**结果：数据全部落地**（不含屏障感知）

| 指标 | R/G | 传统 | 说明 |
|------|-----|------|------|
| min（L\*=7, κ=1） | ≈ 2.0 μs | ≈ **1.4 μs** | 传统免 RTT 的优势占比升至 ~30% |
| mean（L\*=8） | ≈ 2.2 μs | ≈ 1.7 μs（κ≈1.1） | 短批量下浅队列，κ 略有利于传统 |
| max/p99（L\*=10） | ≈ 2.6 μs | ≈ 2.4 μs（κ≈1.4） | 两者接近 |

**结果：完整 cursor step**（数据落地 + 全网确知完成）

| 指标 | R/G | 传统 | R/G 净收益 |
|------|-----|------|-----------|
| **min** | ≈ **2.4 μs** | ≈ 2.9 μs（屏障 1.5 μs） | ≈ 0.5 μs |
| **mean** | ≈ **2.6 μs** | ≈ 3.7 μs（屏障 2 μs） | ≈ 1.1 μs（~30%） |
| **max / p99** | ≈ **3.0 μs** | ≈ 5.4 μs（屏障掉队 3 μs） | ≈ 2.4 μs（~45%） |
| step 间抖动 | ≈ 0.15～0.25 μs | ≈ 1～1.5 μs（屏障项不随 M 缩小） | ~6× |

**两组对照的结论**

1. **批量越小，R/G 的相对优势越大**（就完整 cursor step 而言）：mean 领先从 ~25%（64 token）升至 ~30%（16 token），p99 领先从 ~40% 升至 ~45%。原因是两方案的固定开销此消彼长——串行化时间缩短后，传统方案最大的固定项（软件屏障 1.5～3 μs，**不随 M 缩小**）权重上升，而 R/G 的固定项（RTT + SYNC ≈ 1.0 μs）本来就更小。
2. **若只看数据落地，批量越小传统越占便宜**：RTT 0.6 μs 在 2 μs 的总时长中占比 ~30%，min/mean 情形传统快 0.5 μs 左右。这再次说明对比口径必须取完整 BSP 语义（§8.4 第 1 条）——步与步之间必须确知完成才能推进，"落地但不知道落地"的时间对 BSP 程序不可用。
3. **抖动优势进一步扩大**：R/G 的 step 间抖动只含负载随机项（随 M 减小而减小），传统方案的抖动由屏障与拥塞主导（不随 M 减小），两者比值从 ~5× 升至 ~6×。
4. **短步长的流水线优化**：连续 BSP 步之间，R/G 可用 cursor 超前窗口 W_c（§2.9（1））把世代 v+1 的 request/grant 往返与世代 v 的数据传输重叠，稳态下 RTT 从每步成本中完全消失——短步长（本组场景）正是这一优化收益最大的场景；传统方案的软件屏障处于关键路径上，无法用同样方式隐藏。

### 8.7 第三组对比：每发送方 1024 token（长步长场景）

将每发送方的 token 数升到 1024（总量 131072 grain），其余条件与 §8.3 相同。这一组考察**固定开销被摊薄、串行化与拥塞行为主导**时的相对表现。

**负载随机性**

每条下行链路负载 L ~ Binomial(131072, 1/1024)：均值 **128** grain，标准差 ≈ 11.3。1024 条链路取 max：

> E[L\*] ≈ 168～170 grain；较好情形（p5）≈ 163；较差情形（p99）≈ 177

大样本自平均效应：相对不均衡 L\*/均值降至 ≈ **1.32 倍**（16 token 场景为 4 倍，64 token 为 2.3 倍）。发送方上行为确定的 128 grain/口（18.3 μs），低于下行热口，瓶颈在下行侧。

**一个新出现的分化点：交换机缓冲需求**

长步长下传统方案的队列行为发生质变：全体发送方以上行线速持续注入约 18.3 μs，最热下行口收到 ≈ 169 grain 但排空需 ≈ 24.2 μs——**峰值队列 ≈ 169 − 128 ≈ 41 grain ≈ 287 KB（单口）**，且该值随 M 线性增长；多个半热端口叠加时共享 MMU 吃紧，触发背压/PFC 的风险随 M 上升（本组仍按优待假设不计其代价）。R/G 的出口积压则与 M 无关，恒为 σ ≈ 1～2 grain（≈ 14 KB）——**传统方案缓冲需求 O(M)，R/G 为 O(1)**。

**结果：数据全部落地**（不含屏障感知）

| 指标 | R/G | 传统 | 说明 |
|------|-----|------|------|
| min（L\*=163, κ=1） | ≈ 24.3 μs | ≈ **23.7 μs** | RTT 差距占比已 <3% |
| mean（L\*=169） | ≈ 25.2 μs | ≈ 27.0 μs（κ≈1.1） | 拥塞放大开始反超 RTT |
| max/p99（L\*=177） | ≈ 26.5 μs | ≈ 34～35 μs（κ≈1.35） | 放大系数作用于大基数 |

**结果：完整 cursor step**（数据落地 + 全网确知完成）

| 指标 | R/G | 传统 | R/G 净收益 |
|------|-----|------|-----------|
| **min** | ≈ **24.7 μs** | ≈ 25.2 μs（屏障 1.5 μs） | ≈ 0.5 μs |
| **mean** | ≈ **25.6 μs** | ≈ 29.0 μs（屏障 2 μs） | ≈ 3.4 μs（~12%） |
| **max / p99** | ≈ **27 μs** | ≈ 37～38 μs（屏障掉队 3 μs） | ≈ 10～11 μs（~28%） |
| step 间抖动 | ≈ 0.4～0.5 μs（负载项随 M 增大） | ≈ 2.5～4 μs（κ 波动作用于大基数） | ~7× |

**三组场景纵览**（完整 cursor step）

| 每发送方 token 数 | R/G mean / p99 | 传统 mean / p99 | mean 领先 | p99 领先 | 主导差异来源 |
|------|------|------|------|------|------|
| 16 | 2.6 / 3.0 μs | 3.7 / 5.4 μs | ~30% | ~45% | 固定项：软件屏障 vs 内建 SYNC |
| 64 | 4.0 / 4.8 μs | 5.3 / 7～8 μs | ~25% | ~40% | 固定项 + 拥塞放大各半 |
| 1024 | 25.6 / 27 μs | 29.0 / 37～38 μs | ~12% | ~28% | 乘性项：拥塞放大 κ 作用于大基数；缓冲 O(M) vs O(1) |

**第三组的结论**

1. **固定开销之辩在长步长下自然消解**：request/grant 的 0.6 μs 在 25 μs 的步长中占 2.4%，即使只看数据落地，传统方案的 min 优势也缩到 3% 以内；mean 起传统方案已被拥塞放大反超。
2. **R/G 的优势从"省固定项"转为"省乘性项"**：κ 作用于 L\*×τ_g 这个随 M 线性增长的基数上，绝对收益随步长增大（mean 净收益 1.1 → 1.3 → 3.4 μs；p99 净收益 2.4 → 2~3 → 10~11 μs）——相对百分比下降，但绝对时间扩大。
3. **缓冲需求是长步长下的定性分水岭**：传统方案热口峰值队列随 M 线性增长（本组已达 ~300 KB/口），逼近共享 MMU 极限后将触发背压级联，κ 不再是 1.1~1.35 而是不可控——优待假设在长步长下越来越不成立；R/G 的 σ 级缓冲上界与 M 无关，这正是 §2.10 (σ, ρ) 论证中 ρ ≤ 线速带来的结构性性质。
4. 三组合观：**短步长比固定项，R/G 赢在内建屏障；长步长比乘性项与缓冲，R/G 赢在授权节奏**——两端都占优，且真实 EP 热点流量会同时放大传统方案的两项劣势。

### 8.8 两层网络对比：1024 发送方 × {16, 64, 1024} token

将同样的三组 token 数放到两层网络（§4.1：1024 NPU + 128 Leaf + 64 Spine）重做对比。两层引入的新变量对两个方案的影响**不对称**：

- **传统方案**：路径从 2 hop 变 4 hop，且出现两级路径选择（上行 Leaf→Spine 的 ECMP 哈希 + 目的侧入口选择）。静态哈希的失衡**不随时间平均**（流绑定路径），Spine 中段成为新的冲突层，拥塞放大系数升至 **κ ≈ 1.2～1.5（mean）/ 1.5～2（p99）**；软件屏障的参与者从 128 升到 1024 节点，成本升至 **3～6 μs**。
- **R/G**：固定项温和增加——RTT_rg ≈ 1.1 μs（§4.4）、末 grain 4 hop ≈ 0.9 μs（§4.6）、cursor 两级汇聚 SYNC ≈ 1.2 μs（§4.9）；路径三个自由度仍由知情角色钉扎（§5.2），中段无哈希失衡，上行统计段的残余 κ ≤ 1.05（§4.3（3））。

**负载随机性**（目的均匀随机；瓶颈仍在目的侧下行口，全网 8192 条下行链路取 max）

| M（token/发送方） | 总量（grain） | 每下行口均值 | E[L\*]（p5 / p99） | 相对不均衡 L\*/均值 | 串行化 mean |
|------|------|------|------|------|------|
| 16 | 16384 | 2 | ≈ 9.4（8 / 11） | 4.7× | ≈ 1.3 μs |
| 64 | 65536 | 8 | ≈ 20.5（19 / 23） | 2.6× | ≈ 2.9 μs |
| 1024 | 1048576 | 128 | ≈ 174（169 / 181） | 1.36× | ≈ 24.9 μs |

（同均值下 L\* 略高于单层，因为取 max 的链路数从 1024 增到 8192；发送方上行恰 M/8 grain/口，均低于下行热口。）

**结果：完整 cursor step**（数据落地 + 全网确知完成）

| M | R/G min / mean / p99 | 传统 min / mean / p99 | mean 领先 | p99 领先 |
|------|------|------|------|------|
| 16 | 4.3 / **4.5** / 5.1 μs | 5.0 / 6.6 / 9～10 μs | ~32% | ~47% |
| 64 | 5.9 / **6.1** / 6.8 μs | 6.6 / 8.6 / 12～13 μs | ~29% | ~46% |
| 1024 | 27.4 / **28.1** / 29～30 μs | 28.1 / 36 / 53～54 μs | ~22% | ~45% |

（若只看数据落地：M=16 时传统 mean 仍快约 0.7 μs（免 RTT）；M=64 时两者打平；M=1024 时 R/G 已领先约 5 μs——交叉点比单层更早到来，因为两层的 κ 更大。）

**两层三组的结论**

1. **R/G 的领先幅度全面高于单层，且 p99 领先不随 M 衰减**（稳定在 ~45%～47%；单层从 45% 衰减到 28%）。原因：两层的 κ 是更大的乘性项，对任何 M 都按比例放大传统方案的串行化时间；而 R/G 通过授权钉扎把中段与下行的 κ 都压回 ≈ 1。
2. **两级 ECMP 是传统方案在两层的独有劣势**：静态哈希把流绑定在路径上，失衡不随时间自愈（p99 κ 达 1.8）；改用逐包喷洒（packet spraying）可缓解失衡但引入乱序重组成本。R/G 天然逐 grain 散布且无乱序代价——token_tag + grain_index 的重组机制本来就是协议的一部分（§2.5）。
3. **屏障成本随规模的分化**：软件屏障从 128 节点的 1.5～3 μs 涨到 1024 节点的 3～6 μs（O(N) 通知量 + 更深的树 + 更大的掉队者偏斜）；R/G 的两级汇聚 SYNC 只从 ≈ 0.4 μs 涨到 ≈ 1.2 μs（128 分片 + 一次广播，§4.9）。规模越大，内建屏障的优势越大。
4. **缓冲分化在两层加剧**：传统方案在上行、Spine 下行、Leaf 下行三段都出现随 M 增长的队列（M=1024 时热口峰值 ≈ 46 grain ≈ 320 KB，还未计中段）；R/G 三段各有与 M 无关的硬上界（≈ 12 / 4 / 11 grain，§4.5）。
5. 与单层结论合并：**网络层数越多、规模越大、步长越长，传统方案的乘性劣势（κ、缓冲、屏障）越突出，而 R/G 的固定开销占比越低**——R/G 的架构收益随系统规模单调放大，这与第 6 章"分片资源随所辖出口数而非全网规模增长"的扩展性结论互为印证。
