1 结论速览（TL;DR）

在「1 交换机 + 16 节点 + 200 Gbps」拓扑下，对 DeepSeek-V4-Pro 单层 MoE（EP=16、batch=32、top-6、FP8 dispatch 7 KB / BF16 combine 14 KB per token-expert）做事件级仿真：

  1. incast 是硬墙，且不可优化。热点专家把大量 token 汇聚到单一 rank，使该 rank 的单条 200 Gbps 链路串行化成为时延下界。dispatch 受入向下行链路约束、combine 受出向上行链路约束。该 incast 串行时延占网络时延的绝大部分（ρ_h=0.5 时 dispatch ≈ 370 µs、combine ≈ 740 µs），任何调度都无法突破。
  2. 关键路径上可优化的动态时延上限很小。基线相对 incast 下界仅高出约 4–8%（dispatch 24.6 µs / combine 29.0 µs @ ρ_h=0.5）。SHMEM-POP 把它压到距下界 <1.1 µs（≈O(RTT)），吃掉 95–97% 的可优化空间。
  3. 真正大块的可优化动态时延藏在「拥塞扩散」里。基线缺少接收端配速 + VoQ 隔离，热点输出口缓存填满后 CBFC 反压 + FIFO 队头阻塞让冷 rank 完成时间从 ~34 µs 被顶到 ~380 µs（约 11×）。SHMEM-POP 的 VoQ 隔离 + ESC 信用配速把冷 rank 拉回 Oracle 水平（~33 µs）。
  4. 多平面是降低 incast 硬墙的唯一物理手段：floor ∝ 1/P。P=1→8 时 dispatch floor 从 371 µs 降到 47 µs（×1/8）；SHMEM-POP 在每个 P 都贴着下界。
  5. 单交换机下 incast 硬墙随 EP 规模线性恶化：floor ∝ N。N=16/64/128 时 dispatch floor = 371 / 1426 / 2572 µs；SHMEM-POP gap 恒为 0.84 µs。规模越大越需要靠加平面而非靠调度来压 incast。
  6. P×N 组合 + 缓存跟随匹配：SHMEM-POP 的价值是「缓存效率」。用恒定 ~98 KB 片上缓存达到与开环基线相同的 floor，而基线达到 floor 所需缓存 ∝ BDP×(N−1)/P（N=128/P=1 需 ~3.1 MB/口，物理不可行）。

3 业务/流量模型（MoE EP）

  • EP = R = 节点数；384 routed experts，每 rank 本地 24 个；top-6；每 rank batch = 32 token。
  • dispatch：FP8，每 token-expert ≈ 7 KB；矩阵 M_disp[i][j] = rank i 路由到位于 rank j 的专家的字节数。
  • combine：BF16，每 token-expert ≈ 14 KB；M_comb[j][i] = M_disp 计数 × 14 KB（反向、字节×2）。
  • 热点（incast）：4 个热点专家落在单一 hot rank（rank 0），吸收占比 ρ_h 的路由命中。ρ_h ∈ {0, 0.3, 0.5, 0.7}。
  • 总量：dispatch ≈ 20.7 MB、combine ≈ 41.4 MB（整个 EP 域单层单步，N=16）。

  • incast floor（解析） = max(busiest_uplink, busiest_downlink) / (200 Gbps × P) + 静态
  • headroom = makespan(Baseline) − floor
  • POP gap = makespan(SHMEM-POP) − floor

matched_buffer(P,N) = BDP × ⌈(N−1)/P⌉      （BDP = 200 Gbps × 1 µs ≈ 24.4 KB）

6 解读与边界

  • 关键路径 headroom 为何这么小？ 非阻塞输出排队交换机里，瓶颈链路满负荷时 makespan ≈ floor，与调度顺序基本无关——瓶颈链路就是物理上限。仿真证实了 incast 部分不可优化。
  • combine 可优化空间更小：incast 在热点 rank 的出向上行链路（单口串行），makespan 由该单口决定 → 几乎纯硬墙。
  • 无损 CBFC 下基线对冷流更毒：无丢包反压让队头阻塞演化成「拥塞树」，冷 rank 完成时间被顶到 ≈热点 floor（~11×）。
  • 多平面降低 floor：incast 串行 /= P，是降低 incast 硬墙的唯一物理手段。
  • 规模放大 incast 硬墙：单交换机下 floor ∝ N，N=128 时已达毫秒级。
  • 缓存跟随匹配揭示 SHMEM-POP 的缓存效率：把片上缓存需求从 O(N/P·BDP) 降到 O(BDP)。
  • SHMEM-POP 价值定位：①关键路径逼近 incast 下界；②消除冷流拥塞扩散；③Push ≤ 1 RTT 反压。

建模假设：交换为输出排队非阻塞；三方案全程无损 CBFC；多平面采用 ideal multi-rail 条带；专家计算 τ_wave 此处不计入纯网络动态时延。
