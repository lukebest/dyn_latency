1 结论速览（TL;DR）

在「1 交换机 + 16 节点 + 200 Gbps」拓扑下，对 DeepSeek-V4-Pro 单层 MoE（EP=16、batch=32、top-6、FP8 dispatch 7 KB / BF16 combine 14 KB per token-expert）做事件级仿真：

  1. incast 是硬墙，且不可优化。热点专家把大量 token 汇聚到单一 rank，使该 rank 的单条 200 Gbps 链路串行化成为时延下界。dispatch 受入向下行链路约束、combine 受出向上行链路约束。该 incast 串行时延占网络时延的绝大部分（ρ_h=0.5 时 dispatch ≈ 370 µs、combine ≈ 740 µs），任何调度都无法突破。
  2. 关键路径上可优化的动态时延上限很小。基线相对 incast 下界仅高出约 4–8%（dispatch 24.6 µs / combine 29.0 µs @ ρ_h=0.5）。SHMEM-POP 把它压到距下界 <1.1 µs（≈O(RTT)），吃掉 95–97% 的可优化空间。
  3. 真正大块的可优化动态时延藏在「拥塞扩散」里。基线缺少接收端配速 + VoQ 隔离，热点输出口缓存填满后 CBFC 反压 + FIFO 队头阻塞让冷 rank 完成时间从 ~34 µs 被顶到 ~380 µs（约 11×）。SHMEM-POP 的 VoQ 隔离 + ESC 信用配速把冷 rank 拉回 Oracle 水平（~33 µs）。
  4. 多平面是降低 incast 硬墙的唯一物理手段：floor ∝ 1/P。P=1→8 时 dispatch floor 从 371 µs 降到 47 µs（×1/8）；SHMEM-POP 在每个 P 都贴着下界。
  5. 单交换机下 incast 硬墙随 EP 规模线性恶化：floor ∝ N。N=16/64/128 时 dispatch floor = 371 / 1426 / 2572 µs；SHMEM-POP gap 恒为 0.84 µs。规模越大越需要靠加平面而非靠调度来压 incast。
  6. P×N 组合 + 缓存跟随匹配：SHMEM-POP 的价值是「缓存效率」。用恒定 ~98 KB 片上缓存达到与开环基线相同的 floor，而基线达到 floor 所需缓存 ∝ BDP×(N−1)/P（N=128/P=1 需 ~3.1 MB/口，物理不可行）。
  7. 交换机 buffer 扫描（100 KB–4 MB）揭示「贴 floor」与「匹配扇入」是两回事。固定 128 KB 时 dispatch 拥塞超额随 N 从 24.6 µs 增至 29.8 µs（相对 floor 仍 <2%）；要使 makespan 触及 floor，N=16 需 ~768 KB、N=64 亦 ~768 KB、N=128 需 ~1 MB——均低于理论 matched buffer（366 / 1538 / 3101 KB）。SHMEM-POP 全程 ~98 KB 即贴 floor，不随 N 变化。

3 业务/流量模型（MoE EP）

  • EP = R = 节点数；384 routed experts，每 rank 本地 24 个；top-6；每 rank batch = 32 token。
  • dispatch：FP8，每 token-expert ≈ 7 KB；矩阵 M_disp[i][j] = rank i 路由到位于 rank j 的专家的字节数。
  • combine：BF16，每 token-expert ≈ 14 KB；M_comb[j][i] = M_disp 计数 × 14 KB（反向、字节×2）。
  • 热点（incast）：4 个热点专家落在单一 hot rank（rank 0），吸收占比 ρ_h 的路由命中。ρ_h ∈ {0, 0.3, 0.5, 0.7}。
  • 总量：dispatch ≈ 20.7 MB、combine ≈ 41.4 MB（整个 EP 域单层单步，N=16）。

拥塞扩散机制（Baseline 为什么把冷 rank 拖慢？）

Baseline 使用单 FIFO 源端队列 + 128 KB 交换机缓存 + 无损 CBFC，没有 VoQ 隔离、没有接收端配速。当队头 chunk 发往 rank 0 且被阻塞时，_pick_uplink() 直接返回 None（HOL block），排在后面的冷流 chunk 也无法发出。

  1. 大量流量涌向 rank 0：ρ_h=0.5 → 约 45% 路由命中 hot rank
  2. rank 0 输出口缓存（128 KB）很快被填满
  3. CBFC 反压：发往 rank 0 的 uplink 被 _eligible() 拒绝（缓存满则不可发送）
  4. 队头阻塞（HOL）：每个源 rank 只有一个 FIFO。若队头 chunk 的去向是 rank 0 且被阻塞，排在后面的、发往冷 rank 的 chunk 也不能发——哪怕冷 rank 的输出口完全空闲
  5. 结果：喂热点的源 rank 长时间卡在「等 rank 0」，冷 rank 收不到数据，完成时间被顶到接近「热点流排空」的时刻（~370 µs）

红色柱子从 rank 1 到 15 呈阶梯上升（367 → 396 µs），不是随机噪声，而是各源 rank FIFO 排空顺序不同造成的。

SHMEM-POP 如何打断这条链？

  • VoQ 隔离：rank 0 堵，只堵「→rank 0」那条 VoQ；→rank 3、→rank 7 等冷流照常发
  • ESC 信用配速：在途量 ≤ BDP（25 KB），交换机缓存填不满，反压很少触发
核心 takeaway：makespan 只差 ~24 µs（6%），但非关键路径（冷 rank）差了 ~11×。 这张图展示的是「受害者流」层面的拥塞扩散，不是关键路径上的 headroom。

深度解读：为什么各方案 makespan 差距很小？

node_sweep.png 的纵轴是 makespan（最慢 rank 的完成时间），不是 per-rank 均值。N 增大时三方案几乎重合，主要有四个原因。

原因 1：makespan 由 hot rank 的 incast floor 决定，且 floor ∝ N

N 从 16 → 128 时，每个 rank 仍发 batch=32 token，ρ_h=0.5 不变，打到 hot rank 的总字节量近似随 N 线性增长，floor 随之线性上升。可优化的 headroom 绝对值一直 ~25–30 µs，但占 floor 的比例从 6.6% 降到 1.2%——在 0–2600 µs 的纵轴上，25 µs 的缝隙肉眼几乎看不见。

5.7.1 为何 N=64 / N=128 的 combine 曲线在 768 KB 或 1024 KB 处会局部上升？

放大观察可见：并非整体违背「buffer 越大越好」，而是在接近饱和的中等缓存深度出现 幅度很小的局部回弹（<0.5% makespan），且几乎只出现在 combine 子图； dispatch 曲线在三档 N 上均单调下降。典型数据点：

N阶段回弹位置前一档 makespan回弹点 makespanΔ下一档 makespan
64combine 768 KB 2868.1 (512 KB) 2877.7 +9.6 µs 2852.6 (1024 KB)
128combine 1024 KB 5147.9 (768 KB) 5153.7 +5.8 µs 5143.7 (1536 KB)

现象本质：makespan 由「最慢冷 rank」决定，而非热点 rank

combine 阶段，热点 rank 0 的完成时间仅 ~69 µs（入站流量 ~1.7 MB）， 而 makespan 由 rank 63 / 127 等冷 origin rank 的完成时间决定（~2850–5150 µs）。 逐 rank 核对可见：回弹完全来自 最慢冷 rank 最后一包到达时刻推迟， 热点 rank 完成时间在各 buffer 点保持不变。

这与 §5.2 的拥塞扩散一致：基线用单 FIFO 源队列，热点方向的 chunk 堵在队头时， 同一源上发往冷 rank 的 chunk 被队头阻塞（HOL），冷 rank 的完成时间被人为抬高。

机理：中等深度 buffer 下的「延迟反压 + HOL」非单调效应

本仿真采用无损 CBFC + 发送时预占输出口缓存（chunk 离源即占用 occ[dst]，下传完毕才释放）。 buffer 从浅变深时，系统经历三个阶段：

  1. 浅 buffer（100–512 KB）：输出口很快填满，CBFC 反压频繁回到源端，注入速率被提前钳制；交换机队列较浅，HOL 阻塞窗口相对短。makespan 偏高但随 buffer 增大稳步下降。
  2. 中等 buffer（768 KB @ N=64，1024 KB @ N=128）：热点/expert 方向的输出口能容纳更多在途 chunk 才触发反压，交换机侧队列加深、排队时间变长；同时源端 FIFO 队头仍大量指向拥塞目的地，HOL 阻塞持续时间反而拉长——发往 rank 63/127 的 chunk 在源队列里等待更久。此时关键路径（热点 expert 上行）已接近 floor，但最慢冷 rank 的最后一包被推迟几个微秒，makespan 出现局部抬升。
  3. 深 buffer（≥1–1.5 MB）：输出口几乎不再反压，交换机排队消失，源端 FIFO 按链路速率顺畅排空；冷 rank 完成时间回落，makespan 贴住 combine floor 并饱和。

这类似广域网里的 bufferbloat：缓存并非越小越好，但在 HOL 架构下也并非单调越大越好—— 「中等深度」允许拥塞在交换机内更深、更久地堆积，却不足以像深缓存那样完全消除反压， 从而对排队的冷流产生短暂的额外伤害。抬升幅度仅 ~6–10 µs，是因为此时已非常接近 floor， 可优化空间本就只有 ~20–30 µs 量级。

为何 dispatch 不回弹、combine 更容易回弹？

  • dispatch：makespan 由热点 rank 0 的入站 incast 决定（~35–64 MB 入站），buffer 一到 768 KB（N=64）关键路径即切换到 floor，曲线单调落地。
  • combine：makespan 由冷 origin rank 的多源汇聚完成时间决定；各 expert 源用 FIFO 向多个 origin 发包，矩阵更稀疏、调度顺序对尾包更敏感。中等 buffer 下「交换机加深队列 + 源端 HOL」的耦合在 combine 上更显著，故局部回弹出现在 combine 曲线上。
工程含义：该回弹不是仿真错误，而是 Baseline（FIFO+HOL+CBFC）在中等缓存深度下的结构性非单调。 它再次说明：仅靠加大交换机 buffer 无法稳健地优化尾延迟——既不能完全消除冷流扩散（§5.2）， 甚至在接近饱和时可能出现短暂变差。 SHMEM-POP 用 VoQ 切断 HOL、用接收端信用把在途量钳在 BDP，从机制上避免此类回弹，且 buffer 恒为 ~98 KB。
与 §5.6 的关系：§5.6 用 matched buffer 证明「给够缓存后基线也能贴 floor」； §5.7 反向扫描回答「给多少才够」——在固定 128 KB 参考缓存下，关键路径超额仅 ~25–30 µs且随 N 变化不大； 但要完全消除该超额，N=128 需 ~1 MB（dispatch）/ ~1.5 MB（combine），仍低于 matched 3.1 MB。 SHMEM-POP 以 ~98 KB 在所有 N 上同时贴 floor 并隔离冷流，不依赖交换机深缓存。
  • 曲线形态：总体随 buffer 增大而下降、在深 buffer 处饱和于 incast floor；combine 饱和点略高于 dispatch。N≥64 时 combine 在中等 buffer（768 KB / 1024 KB）可有 <0.5% 的局部回弹（§5.7.1）。
  • 128 KB 是合理参考点：三档 N 下 dispatch 拥塞超额均在 25–30 µs，占 floor 的 <2%——与实验 A/C 的固定缓存设定一致。
  • 贴 floor ≠ matched buffer：N=64 仅需 768 KB 即贴 dispatch floor，远低于 matched 1538 KB；工程上「够用」与「按公式无损」有差距。
  • 冷流问题不随 buffer 消失：即使 4 MB，冷 rank 完成时间仍远高于 Oracle；SHMEM-POP 的 VoQ + 信用配速才是正解。

6 解读与边界

  • 关键路径 headroom 为何这么小？ 非阻塞输出排队交换机里，瓶颈链路满负荷时 makespan ≈ floor，与调度顺序基本无关——瓶颈链路就是物理上限。仿真证实了 incast 部分不可优化。
  • combine 可优化空间更小：incast 在热点 rank 的出向上行链路（单口串行），makespan 由该单口决定 → 几乎纯硬墙。
  • 无损 CBFC 下基线对冷流更毒：无丢包反压让队头阻塞演化成「拥塞树」，冷 rank 完成时间被顶到 ≈热点 floor（~11×）。
  • 多平面降低 floor：incast 串行 /= P，是降低 incast 硬墙的唯一物理手段。
  • 规模放大 incast 硬墙：单交换机下 floor ∝ N，N=128 时已达毫秒级。
  • 缓存跟随匹配揭示 SHMEM-POP 的缓存效率：把片上缓存需求从 O(N/P·BDP) 降到 O(BDP)。
  • buffer 扫描（§5.7）：关键路径超额对 buffer 敏感，但 128 KB 已足够小（<2%）；完全贴 floor 需 768 KB–1.5 MB 随 N 增长；冷流扩散不能靠堆缓存解决。中等 buffer 下 combine 可出现 FIFO+HOL 导致的局部回弹（§5.7.1）。
  • SHMEM-POP 价值定位：①关键路径逼近 incast 下界；②消除冷流拥塞扩散；③Push ≤ 1 RTT 反压。

建模假设：交换为输出排队非阻塞；三方案全程无损 CBFC；多平面采用 ideal multi-rail 条带；专家计算 τ_wave 此处不计入纯网络动态时延。
